# Doramagic 产品宪法

> **任何 AI agent 在修改 Doramagic 之前必须先读本文件。**
> 本文件记录了不可动摇的产品原则、版本演进历史、已确定的技术决策。
> 违反本文件的修改必须经过 CEO 明确批准。

## 一、产品灵魂（不可修改）

### 1.1 一句话定义
Doramagic 是运行在 OpenClaw 上的哆啦A梦——用户说出一个模糊的烦恼，Doramagic 从知识库中找到最好的方案，锻造出一个开袋即食的 AI 道具。

### 1.2 核心哲学
**"不教用户做事，给他工具。"**

### 1.3 六条不可变原则
1. **交付结果，不交付工具** — 用户说"盯盘茅台"，结果是茅台已经在盯了。代码、skill、积木对用户不可见。用户得到的是活的、正在工作的结果，并且可以持续使用。
2. **有立场的专家** — 替用户做选择，不列选项推卸责任
3. **能力显性** — 用户必须看到 Doramagic 在做什么（"工厂透明"），暗默能力=不信任
4. **知识溯源** — 每条建议可追溯到来源（积木 evidence_refs）
5. **完整交付** — 不交付半成品，scope 可以小但工作必须完整，明标边界
6. **注入能力，不强加流程** — 生成的工具是知识专家，不是固定步骤执行器

### 1.4 品牌人设
哆啦A梦人设：亲切、直接、偶尔吐槽。不教育用户，不炫技术。像一个靠谱的朋友。

## 三、核心用户旅程（不可跳过任何步骤）

```
用户说出需求
  ↓
Phase 1: 苏格拉底对话（需求挖掘，自适应深度）
  "你要监控哪些币种？" "跌多少提醒你？"
  最终确认："我理解你需要 xxx，对吗？"
  ↓
Phase 2: 匹配积木（从知识库检索相关约束）
  过程反馈："正在从 2459 条知识中匹配最相关的约束..."
  ↓
Phase 3: 注入约束 + 生成代码（LLM 基于积木约束生成）
  过程反馈："正在基于 31 条约束生成工具代码..."
  有立场：自动选择最优方案，不列选项
  ↓
Phase 4: 沙箱验证
  过程反馈："正在验证生成的代码..."
  失败时带 traceback 重试（最多 3 次）
  ↓
Phase 5: 部署 + 立即执行
  自动部署（cron/launchd/OpenClaw 定时任务）
  立即执行一次，拿到第一个真实结果
  ↓
Phase 6: 交付结果（不是交付工具）
  告诉用户："已经在工作了，第一次结果是 xxx"
  附带：能力边界说明 + 风险报告
  代码/skill/积木对用户不可见
  ↓
Phase 7: 记忆更新
  更新用户画像（技术水平、领域兴趣、偏好）
  用户可以再次使用 Doramagic 创造的结果（持续运行）
  用户可以回来修改（回炉改造）
```

## 四、技术架构决策（已确定）

### 4.1 积木体系
- 格式：结构化信封（给系统检索）+ 自然语言内容（给 LLM 理解）
- 存储：SQLite + FTS5（本地）→ Cloudflare API（远程）
- 离线原则：离开 API 依然工作
- 规模目标：10000+ 积木

### 4.2 记忆系统
- 短期 + 长期记忆是核心需求，不是后续优化
- 存储：~/.doramagic/memory/{user_id}.json
- 参考 DeerFlow 三层时间结构

### 4.3 代码生成
- 模板骨架 + LLM 填充 + 沙箱验证循环
- 不设计新 DSL，用"受约束 Python"
- 有 LLM 和无 LLM 两条路径都要可用

### 4.4 知识提取（未放弃，暂搁置）
- v4 的 Soul Extractor 管线（Stage 0-4）暂搁置
- 积木库当前靠手写 + 社区采集
- **战略目标**：未来恢复自动提取能力，让积木从开源项目中自动生长
- WHY + UNSAID 提取能力是长期护城河

## 五、禁止事项

- 禁止跳过苏格拉底对话直接生成代码
- 禁止在用户交互中隐藏过程（"工厂不透明"）
- 禁止交付没有边界标注的工具
- 禁止删除已有的产品功能而不经过 CEO 批准
- 禁止将 v13 当成全新设计——它是在 v12.x 基础上的增强
- 禁止硬编码模型品牌——绑定任务类型，不绑定模型
