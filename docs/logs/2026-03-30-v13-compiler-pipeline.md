# Doramagic v13 个性化编译器建设
日期: 2026-03-30
执行者: Claude Code opus + 多 sonnet 子代理

## 总结

今天完成了 Doramagic 从"知识提取工具"到"个性化编译器"的完整转型，包括产品重新定义、积木体系建设、主管道开发。

## 完成的工作

### 1. 产品方向重定义

通过 `/office-hours` YC Startup 模式确定新方向：
- 核心价值：生成可用的个性化工具，不是提取设计智慧
- 废弃实时 GitHub 搜索
- 交付物 = live outcome（活的结果），不是 SKILL.md 知识包
- Codex 第二意见确认："personalization compiler for AI terminals"

### 2. 全面技术调研

- 10 个子代理并行调研（AI 知识表示、个性化、代码生成、工作流编排、市场现状等）
- DeerFlow 源码深度分析（Opus 级别，5 个模块）
- Claude Code/OpenClaw/Cursor 知识注入机制对比
- 三个关键分歧解决：结构化信封 + 自然语言内容、双观众（系统+LLM）、10000+ 规模

### 3. 积木 v2 体系建设

| 组件 | 数量 |
|------|------|
| BrickV2 Schema | Pydantic 模型 |
| BrickStore | SQLite + FTS5 + 离线回退 |
| 场景积木 | 25 个（3 手写 + 22 分类） |
| v1 迁移积木 | 21 个（460 约束 + 148 失败模式） |
| API 目录积木 | 52 个（1436 个公开 API） |
| 社区踩坑经验 | 160+ 条 |

| 失败模式总数 | 424 条 |

### 4. A/B 测试验证

测试用例："监控比特币价格，跌10%提醒我"
- 方案 A（裸 LLM）：196 行，基础错误处理，无 Telegram 专项错误处理，同步架构
- 方案 B（积木约束）：305 行，11 处限流处理，7 处 Telegram 错误处理，异步架构
- **结论：积木约束让代码从"能跑的 demo"变成"生产级工具"**

### 5. 主管道建设

- PersonalizationCompiler：8 步编译流程（意图→积木→约束→代码→验证→记忆）
- MemoryManager：短期+长期用户画像，跨会话个性化
- doramagic_compiler.py：新入口脚本
- SKILL.md：新增 Compiler Mode v13

### 6. 基础设施修复（上午）

v12.3.1 → v12.3.2 → v12.3.3 连续发布，修复 14 个基础设施问题。
7 次 OpenClaw/Telegram 真实测试。

## 当前已知限制

### 主管道不调用 LLM

当前 `llm_adapter=None`，三个环节全走降级模式：
- 意图解析：正则匹配（不如 LLM 准确）
- 代码生成：25 行模板骨架（不如 LLM 生成的 305 行完整代码）
- 记忆更新：关键词计数（不如 LLM 提炼画像）

积木匹配和约束注入正常工作（31 条约束已注入），但没有 LLM 消费这些约束。

### 需要下一步解决

1. 接入 LLM API（Anthropic/OpenAI/MiniMax）到编译器管道
2. 在 OpenClaw 中部署并测试 Compiler Mode
3. 社区知识持续采集（目前 160 条，每分类平均 7 条，目标 50+）
4. v12.3.4 发布（含 v12.3.3 后的所有修复）
5. Cloudflare API 层（Phase 2）

## 关键决策记录

| 决策 | 选择 | 原因 |
|------|------|------|
| 产品定位 | 个性化编译器 | 用户需要工具不是知识 |
| 积木格式 | 结构化信封 + 自然语言内容 | 双观众：系统检索 + LLM 理解 |
| 记忆系统 | 核心需求，同步建设 | CEO 明确："短期和长期记忆是 AI 产品非常重要的" |
| 存储方案 | SQLite + FTS5 → Cloudflare API | 离开 API 依然工作 |
| 全球视角 | 积木面向全球用户 | CEO 明确：不只做中国市场 |
| API 全量入库 | 1436 个全部入库 | CEO 明确：为什么不能全部作为知识库 |
