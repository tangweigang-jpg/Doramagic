# SOP v1.2 四方评审汇总

**日期**：2026-04-06
**评审方**：Grok / Gemini / GPT / Sonnet (Claude)

---

## 一、总体判断对比

| 评审方 | 一句话 |
|--------|--------|
| **Grok** | 结构严谨但仍是"金融优化版 SOP"，直接用于 Next.js/dbt 会产生大量摩擦 |
| **Gemini** | 实战鲜血洗礼的防御性工程杰作，但过度耦合"一次性顺序执行"假设 |
| **GPT** | 已从"写文档"转向"编译可执行工件"，但离跨行业可复用还有一层抽象没做完 |
| **Sonnet** | 知识中心主义转型正确完成，三维压缩是创新，但金融渗透比团队自己意识到的更深 |

**四方共识：方向正确，但金融烙印过深，需要一层领域无关的抽象。**

---

## 二、Q1-Q7 四方共识与分歧

### Q1：通用性

**4/4 一致**：金融特有元素深度渗透，不只是例子问题，判断标准本身就是金融语言。

| 改进方向 | Grok | Gemini | GPT | Sonnet |
|---------|------|--------|-----|--------|
| Spec Lock 判断标准改为领域无关元准则 | ✅ | ✅ | ✅ | ✅ |
| 引入 Crystal Archetypes / 形态分类 | — | ✅ | — | — |
| B/BA/DK/RC 降级为可插拔分类器 | — | — | ✅ | ✅（需跨领域语义说明） |
| Intent IR / Pattern schema | — | — | ✅ | — |

**Sonnet 独特贡献**：Spec Lock 应改为三条元准则——(a) AI 无约束时会自主选择的参数；(b) 不同选择导致结果不可比较的参数；(c) 与用户意图直接绑定的参数。这三条在任何领域都成立。

---

### Q2：Stage 结构

**4/4 一致**：四阶段对"运行脚本"适用，对"搭建应用/迭代构建/长时间训练"不适用。

| 建议 | 提出者 |
|------|--------|
| 引入 3 种流控模式：Pipeline / Iterative / Patch | Gemini |
| 通用 5 阶段：prepare→build→verify→stabilize→deliver | GPT |
| 任务类型分类（一次性产出 / 迭代构建 / 长时间运行） | Sonnet |
| Stage 允许 loop 标记 | Gemini + Sonnet |
| 增加 stage_type 字段（script-based vs declaration-based） | Grok |

**CTO 综合判断**：Sonnet 的三类任务分类 + Gemini 的三种流控模式本质上是同一个洞察的两种表达。合并为：

```
类型 A（一次性产出）→ Pipeline 模式：setup → execute → validate → deliver
类型 B（迭代构建）→ Iterative 模式：scaffold → [loop: implement ↔ test] → integrate → deliver
类型 C（长时间/声明式）→ Patch 模式：analyze → generate → verify → apply
```

---

### Q3：知识压缩三维筛选

**4/4 认为框架合理但有遗漏维度**。

| 遗漏维度 | 提出者 | 说明 |
|---------|--------|------|
| **爆炸半径 / Decision Impact** | Gemini + GPT | 出错的后果严重度，一票否决权 |
| **操作化程度 (Actionability)** | Sonnet | 知识是否包含可执行指令（"需谨慎"vs 具体示例） |
| **可执行性 (Enforceability)** | GPT | 是只能提醒还是能 gate/check/hook |
| **时效性** | Grok | 过期风险 |
| **可组合性** | Grok | 与其他知识安全组合 |

**4/4 一致**：AI 新颖度判断过于主观，需要量化锚点。

| 建议 | 提出者 |
|------|--------|
| 建立《模型知识盲区基线登记册》 | Gemini |
| 用标准化测试提示词验证（问 AI，看它答对否） | Sonnet |
| 依赖模型 Training Cutoff 版本号 | Gemini |

---

### Q4：SKILL.md 渲染路径

**核心分歧**：References 加载可靠性。

| 观点 | 提出者 |
|------|--------|
| AI 会声称"我已经读了"然后编造——必须程序化强制 Tool Call | Gemini |
| 必须在执行流程中加入强制验证步骤 | Sonnet |
| references 需要从"附录"改为"有加载触发条件的知识页" | GPT |
| scaffold 只适合文件结构明确的场景，多文件嵌套需要 AST Patch 替代 | Gemini |

**4/4 一致**：当前 references/ 加载假设不可靠，需要强制 read 机制。

---

### Q5：Delivery Gate 可迁移性

**4/4 一致**："必须可程序化验证"过强，需要 Hard Gate + Soft Gate 双层体系。

| Gate 类型 | 适用 | 验证方式 |
|-----------|------|---------|
| Hard Gate | 文件存在、build 通过、exit code 0、指标阈值 | 程序化自动验证 |
| Soft Gate | 架构一致性、代码质量、文档覆盖 | LLM-as-a-Judge 或结构化 self-check |

Gemini 独特贡献：Soft Gate 仍然是程序化的，只是编译器换成了验证 LLM——"打开 Headless Browser 截图，传给视觉模型"。

---

### Q6：时效性管理

**4/4 一致**：这是 SOP v1.2 的**结构性盲点**，是最高优先级修复项。

| 建议 | 提出者 |
|------|--------|
| Crystal IR 加 `valid_until` / `deprecation_notice` | Grok |
| Crystal IR 加 `version_pins` + `review_triggers` | Sonnet |
| 编译时检查 commit hash，超 90 天告警 | Grok |
| Step 1b 强制加 Environment Version Check | Gemini |
| 基于 GitHub Releases webhook 触发重编译 | Gemini |
| 四层（Blueprint/Constraint/UseCase/IR）都加 `source_commit` + `validated_at` | GPT |

---

### Q7：执行可操作性

**4/4 一致**：Step 5（知识压缩）和 Step 1a（Spec Lock 判断）最容易卡住，最依赖隐性经验。

| 补充建议 | 提出者 |
|---------|--------|
| 增加 Decision Handbook（20-30 正反例） | GPT |
| 增加最小 end-to-end compiled example | GPT |
| 增加 Crystal TDD Verification（沙箱验证步骤） | Gemini |
| 增加知识压缩决策日志（筛选理由记录在 IR 中） | Grok |

## 三、四方独有发现

| 评审方 | 独有发现 |
|--------|---------|
| **Grok** | "AI 新颖度"会随模型更新失效——需要与模型版本绑定 |
| **Gemini** | "请阅读 references/"必须改为"强制 Tool Call 读取"——AI 的懒惰和幻觉会导致分层知识被完全忽略 |
| **GPT** | B/BA/DK/RC 应降级为领域可插拔分类器，SOP 主体用通用语义：`behavioral_rule` / `assumption` / `platform_constraint` / `regulatory_constraint` |
| **Sonnet** | 三维筛选遗漏 Actionability（操作化程度）维度——"需谨慎"和"必须用复合键 ['id', 'updated_at']"是完全不同的知识质量 |

---

## 四、优先级排序（四方加权）

| 优先级 | 改进项 | 推荐者 | 理由 |
|--------|--------|--------|------|
| **#1** | **时效性管理** | 4/4 | 无保鲜机制的晶体会变成"毒药指令" |
| **#2** | **References 强制加载** | 3/4 | AI 不会主动读取——必须程序化 |
| **#3** | **Stage 拓扑泛化** | 4/4 | 当前只适合脚本执行，不适合迭代构建 |
| **#4** | **Delivery Gate 双层化** | 4/4 | Hard + Soft gate 覆盖非数值场景 |
| **#5** | **Spec Lock 判断改为领域无关元准则** | 4/4 | 当前金融参数列表不可迁移 |
| **#6** | **知识压缩补 Actionability 维度** | 2/4 | 抽象知识 vs 可执行知识的区分 |
| **#7** | **补 end-to-end 非金融编译示例** | 3/4 | 无示例操作者无锚点 |
