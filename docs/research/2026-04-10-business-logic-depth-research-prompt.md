# 研究提示词：自主 Agent 如何提取深层业务逻辑

> 请独立思考后回答，不要互相参考。回答请用中文。

## 背景

Doramagic 从开源项目源码中提取结构化知识（蓝图 Blueprint + 约束 Constraint）。我们刚构建了一个完全自主的提取 Agent，用 MiniMax-M2.7 通过 tool_use 探索源码、提取知识。

Agent 能跑通完整流程（蓝图提取 → 约束采集），SOP 合规检查全通过。但有一个严重问题：**业务逻辑提取深度不足**。

## 问题具体表现

以 zvt（A 股量化交易框架）为例：

| 指标 | 人工迭代版 (111KB) | Agent 自动版 (29KB) |
|------|-------------------|-------------------|
| stages | 8（语义化：schema_layer, trader_engine, sim_account...）| 6（通用：stage_1, stage_2...）|
| Business Decisions | 56 条 | 40 条 |
| M（数学/模型选择）| 27 条 | **0 条** |
| RC（监管规则） | 6 条 | 1 条 |
| BA（业务假设） | 20 条 | 3 条 |
| T（纯技术选择） | 1 条 | **10 条**（大量该标 M/B 的被标成了 T）|

核心缺失：Agent 不理解"这个设计选择背后有什么数学理论""这是监管强制还是作者选择""这个默认值编码了什么市场假设"。

## 我们的 BD 分类框架（六类）

| 类型 | 判别问题 |
|------|---------|
| T（技术选择）| 改了这个，投资结果会变吗？→ 不会 = T |
| B（业务决策）| 改了这个，系统的交易行为会变吗？→ 会 = B |
| BA（业务假设）| 这个默认值背后假设了什么市场规律？ |
| DK（领域知识）| 换个市场（如美股→A股），这条还成立吗？→ 不成立 = DK |
| RC（监管规则）| 这是作者的选择，还是法规/交易所强制的？→ 强制 = RC |
| M（数学/模型选择）| 这是基于数学/统计理论的选择吗？换了方法精度会变吗？→ 会 = M |

M 类尤其重要：在量化/定价/风控项目中，业务逻辑的本质就是数学模型选择。

## 约束条件

1. **完全自主**：Agent 7×24 运行，无人干预。不能依赖人工迭代来深化分类
2. **无参照蓝图**：新项目首次提取时没有旧版本可参考
3. **不能用预设关键词列表**：不同领域的数学/监管术语差异巨大，预设列表必然不完整
4. **LLM 能力边界**：当前用 MiniMax-M2.7（204K context），非最强模型。结构性的"理解-执行脱节"无法靠 prompt 解决

## 已有研究成果

1. 业务逻辑不在框架代码中，而在 `examples/`、`notebooks/`、`tutorials/` 的具体用例中
2. 正确的粒度是"完整的投资/分析工作流"（如 `macd_day_trader.py`），不是单个函数
3. v9 晶体测试证明：即使蓝图完整，缺少业务逻辑标注的晶体无法驱动 AI 完成端到端交付
4. "理解-执行脱节"是 LLM 结构性限制——理解一条规则 ≠ 执行它

## 核心研究问题

**Q1：自主 Agent 如何在没有人工迭代的情况下，从源码中提取深层业务逻辑分类（尤其是 M/BA/RC）？**

请从以下角度分析：
- 是 prompt 策略问题（怎么问）还是架构问题（怎么组织多轮探索）？
- 是否存在不依赖关键词列表的通用方法来识别数学模型选择？
- "自我追问"（self-questioning）策略是否可行——让 Agent 对自己的标注结果提出质疑？
- 有没有相关的学术研究或工程实践（如代码理解、program analysis、concept extraction）？

**Q2：如果单轮提取深度注定不够，多轮迭代应该怎么设计？**

约束：
- 必须是 Agent 自主迭代，不需要人介入
- 必须有收敛条件（不能无限迭代）
- 必须比"跑两遍一样的 prompt"更聪明

请从以下角度分析：
- Critique → Refine 模式（先生成，再自我批评，再修正）是否适用？
- 专家角色切换（先用"架构师"视角，再用"业务分析师"视角，再用"监管专家"视角）是否比单一视角更有效？
- 如何设计收敛判据（什么时候"够深了"可以停止）？

**Q3：是否需要改变蓝图的结构来适应自主提取的限制？**

当前蓝图要求一步到位把所有 BD 分类好。是否有更好的结构？例如：
- 先提取"原始设计决策"（不分类），再用独立 phase 做分类
- 分类作为"置信度"而非"确定标签"（每个 BD 标注 T=0.2, M=0.6, BA=0.2）
- 接受"粗分类"（T vs 非 T），再对非 T 做细分
