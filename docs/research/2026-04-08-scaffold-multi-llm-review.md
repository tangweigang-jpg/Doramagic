# 多方评审：知识 + 骨架混合模式提案

> 日期：2026-04-08 | 状态：评审完成，待 CEO 决策

## 评审背景

bp-009 v1.9/v2.0 晶体在 OpenClaw + MiniMax-M2.7 上的端到端实测暴露一个结构性问题：**宿主 AI 能理解安全规则但不保证在生成代码中实现**。

实测数据（3 次回测）：
- T+1 交割限制：1/3 实现
- 涨跌停过滤：0/3 实现
- output_validator assert：0/3 实现

提案：在晶体中新增 Scaffold 段，安全检查代码内置，AI 只填策略逻辑。

## Grok 评审

**Q1. 普遍问题还是晶体不够好？**
LLM 的结构性局限，非晶体问题。所有主流模型在长上下文下都存在"理解-执行脱节"：知识被内化成概率分布，但代码生成时倾向走最简路径或"创意重构"。3 种强化手段（FATAL、模板、YAML）都只是提示工程层面的缓解，无法根治。

**Q2. Scaffold 是好方案吗？**
高度认同。核心假设完全正确：把控制权从"模型自主性"转移到"结构化约束"上，符合知识工程中"可执行配方"的本质。比纯知识注入提升一个数量级。

**Q3. 最大风险？**
两点：①AI 直接忽略骨架重新生成代码（高自主性模型概率约 15-25%）；②过度限制灵活性，非常规策略时骨架变"枷锁"。

**Q4. 你的设计方案？**
"混合模式 + 执行时校验"双保险：
- 晶体提供骨架（主路径）+ 纯知识（备选），directive 写死优先使用骨架
- 骨架嵌入运行时自校验函数（validate_backtest_safety()），失败则拒绝输出并回滚
- 增加微型代码审查 Agent，交付前扫描代码是否符合约束

**Q5. 结论**
**有条件采用。** 立即引入 Scaffold 作为默认路径，同时配合运行时自校验和回滚机制。

## GPT 评审

**Q1. 普遍问题还是晶体不够好？**
LLM 的普遍结构性问题。模型可以"理解规则"，但代码生成阶段仍会为完成主任务而漏掉次级约束。Prompt 工程能提高命中率，但很难推到 100%。只要要的是"必做"，最终就得把规则变成可执行结构。

**Q2. Scaffold 是好方案吗？**
对，但只算半个解法。骨架把任务从"自由生成"改成"受限填空"，对刚性规则有效。但模型仍可能绕开、改坏骨架。更稳方案是"骨架 + 自动校验 + 不通过就拒绝交付"。

**Q3. 最大风险？**
产生虚假安全感——看起来用了骨架，但关键语义已漂移。第二是架构假设固化（只适合向量化回测）。第三是维护成本膨胀。

**Q4. 你的设计方案？**
三层控制：
- 第一层：晶体知识和约束（告诉 AI 为什么）
- 第二层：可填充骨架（限制生成空间）
- 第三层：机器可执行校验（检查代码是否真的包含 shift(1)、涨跌停、费用分项）
终极方案：安全规则抽成可导入的运行库或 AST patcher，AI 只能填策略，不碰交易安全核心。

**Q5. 结论**
**有条件采用。** 方向对，但不能只上 Scaffold，必须同时补"不可删除的安全库"或"自动验收器"。Scaffold 是控制面的一个组件，不是最终解。

## 四方共识汇总

| 维度 | Claude Opus | Claude Sonnet | Grok | GPT |
|------|-------------|---------------|------|-----|
| 问题定性 | LLM 结构性局限 | LLM 结构性局限 | LLM 结构性局限 | LLM 结构性局限 |
| Scaffold 方向 | 认同 | 认同 | 高度认同 | 认同（半个解法） |
| Scaffold 够不够 | 不够，需 exec hook | 不够，需 AST 扫描 | 不够，需运行时校验 | 不够，需三层控制 |
| 最大风险 | 晶体膨胀 + 维护成本 | AI 绕过 scaffold | AI 忽略(15-25%) + 灵活性 | 虚假安全感 + 语义漂移 |
| 最终判断 | CONDITIONAL ADOPT | ADOPT with conditions | CONDITIONAL ADOPT | CONDITIONAL ADOPT |

**全票一致：有条件采用。**

共同条件：
1. 骨架仅限 FATAL 级 RC 约束
2. 安全代码为不可替换区域
3. 必须配合验证层（后置校验 / exec hook / AST 扫描）
