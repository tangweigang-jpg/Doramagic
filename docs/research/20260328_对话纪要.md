## 三、Doramagic v12.1.1 管线架构重构设计

### 3.1 Tangsir 提供额外上下文

**Tangsir 提供了 8 条真实问题**：
1. 用户给了 GitHub URL，系统仍跑 Discovery（浪费）
2. 用户请求模糊，不追问直接盲搜（质量差）
3. awesome-list 和深度框架走同一条提取路径（错误）
4. 3 个项目串行提取，一个失败污染全局
5. 编译占总耗时 51%（138s/268s）
6. 编译完才发现质量差，前 4 分钟白费
7. 任何失败 = 整个 run 失败，用户只看到"失败"
8. 执行 4 分钟黑盒无反馈

**Tangsir 提供的 Doramagic 已有资产**：
- **v7.0 三层架构**：Controller（12 状态 FSM）→ Executor（8 个独立执行器）→ Platform Adapter
- **Lease Token 门控**：每个 phase 获取 lease 才能执行，TTL 过期自动降级
- **8 条血泪教训**：
  1. skill name 就是路由 key，改名 = 全部调用断裂
  2. OpenClaw exec 工具不可用，必须用 subprocess
  3. LLM 会逃逸预设角色，必须结构化约束
  4. 控制权不能交给 LLM，必须代码控制流程骨架
  5. 信息饥饿：上游阶段不传足够信息给下游 = 下游输出质量暴跌
  6. 三方研究法（Claude+Codex+Grok）是发现盲点的最有效方式
  7. 砖块注入必须有域隔离，否则无关领域知识污染提取
  8. 必须有端到端冒烟测试

- **外部框架启发**：
  - LangGraph：图状态机、TypedDict+Reducer、Send() API 动态扇出、Checkpointer
  - DSPy：编译范式、Module 声明式、Optimizer 自动调优 prompt
  - CrewAI：任务输出截断是常见 silent failure、无限循环防护
  - LlamaIndex：RouterQueryEngine 按输入类型路由、数据流隔离
  - **Haystack 2.0**：Pipeline.connect() 显式声明数据流、ConditionalRouter 用 Jinja2 做条件路由、支持 Fan-out/Fan-in/Loop

### 3.2 G大脑袋输出 v12.1.1 架构设计方案

**核心改造**：确定性上下文路由（Deterministic Context Routing），基于 `PipelineContext` 动态路由，替代固定线性 TRANSITIONS。

**7 大设计模块**：

1. **输入路由器（Input Router）**：4 象限分类（URL 直达/明确项目名/泛领域探索/意图模糊），确定性路由标志，模糊意图触发 CLARIFY 状态暂停管线请求用户补充

2. **状态机改造与条件边**：Stage 0 后加入判定（Markdown 占比 > 80% → `is_collection=True`），awesome-list 类项目路由到轻量级元数据提取路径

3. **扇出/扇入与数据隔离**：Fan-out：N 个独立 `CD_Executor` 子进程，独立 staging 目录。Fan-in：只要 ≥1 个成功就继续，失败记录到 `context.failed_repos`

4. **信息传递契约**：严格定义上下游 Payload Schema（A→B 传 `SearchConstraints`，B→CD 传 `RepoContext`，CD→E 传结构化卡片+DSD 分数）

5. **质量反馈环与模块化编译**：Pre-flight Check（E 结束时规则引擎快检，不达标直接降级不进入高耗时 F）。针对性修复：G 报错指出 `weakest_module`，仅将该模块发回 F_PARTIAL 重新编译

6. **降级交付体系**：Level 1（最佳）/ Level 2（编译失败降级到原始卡片库）/ Level 3（提取部分失败降级到独立报告）/ Level 4（发现失败降级到项目清单）

7. **进度事件架构**：`run_events.jsonl` Schema，Platform Adapter 每 3-5 秒向 Telegram 推送状态更新
