## 1. 分类质量

### 整体评价

23 条 business_decisions 的分类体系是整个升级的核心价值所在。总体而言，分类逻辑自洽，rationale 对 AI agent 友好（给出了"为什么"而非仅"是什么"）。

### 准确无误的分类（代表性条目）

- **先卖后买 → B**：正确。这是作者在无杠杆假设下的主动选择，非技术约束。
- **T+1 信号延迟 → RC**：经过四方评审校正后已正确。原错误分类为 B，校正为 RC 是关键一步。
- **印花税 → RC**：正确，且附带了税率历史演变，信息密度高。
- **turnover_rate 作为一等字段 → DK**：精准。这是 A 股量价文化的编码，是真正的领域知识。
- **涨跌停、T+1 持仓约束 → RC**：正确，监管规则范畴。

### 可争议或需讨论的分类

**1. 止损 -30% 标注为 "B/BA" 复合类型**

当前处理方式是用斜杠写 `"B/BA"`，理由是"在多种方案中主动选择了 -30%（B）且该值偏宽松是教学取向假设（BA）"。

评议：这个处理思路正确，但 AI agent 读到 `"B/BA"` 时可能不清楚主从关系。建议改为：
```yaml
type: BA
subtype: B  # 同时是业务决策（选了具体值），主导分类是业务假设
```
或者在 rationale 中明确："主分类 BA（教学取向假设），次分类 B（可替换的具体值）"。

**2. 回测与实盘共用主循环 → "T/B"**

rationale 写道"T：工程统一实现；B（正向副产物）：保证回测与实盘一致"。

评议：这里的逻辑值得商榷——"回测与实盘一致"在量化领域不只是"正向副产物"，更是一个积极的**业务决策**（避免实盘与回测的行为差异）。建议主分类改为 B，将 T 作为实现手段注释，而非并列。

**3. 每种标的×level×adjust = 独立 Schema → "T/BA"**

评议：这个复合分类是合理的。但 rationale 中的 BA 部分（"隐含假设复权方式不同 = 本质不同的数据"）实际上更接近 DK（领域知识）——A 股的复权方式之所以是独立数据集，是因为 A 股的复权机制和除权除息文化导致前复权/后复权/不复权数据在技术指标上有根本差异，这是领域知识而非可以轻易替换的假设。

建议：`type: "T/DK"` 比 `"T/BA"` 更准确。

**4. 停牌处理 → "B/DK"**

评议：停牌在 A 股有明确的交易所规则（停牌需公告，有时间限制），部分场景属于 RC。但核心分类 DK 是准确的（停牌的业务含义和对策略的影响是领域知识）。可以考虑补充 `subtype: RC`。

**5. 除权除息处理 → "B/BA"**

评议：这个分类偏低。除权除息是交易所强制执行的结算规则（送股/分红的价格调整是交易所规定），应包含 RC 分量。当前标注为纯 B/BA 忽略了监管层面的强制性。建议改为 `"RC/B/BA"`，或至少在 rationale 中补充"除权除息日由交易所强制执行"。

### 6 个 known_gap 的 severity 评级合理性

| gap | 当前 severity | 评审意见 |
|-----|--------------|---------|
| 涨跌停 | critical | 正确。动量策略回测虚高是核心问题 |
| 除权除息 | critical | 正确。年化偏差 5~15pp，且系统性正偏 |
| 停牌 | high | 正确。长期停牌时 fill_gap 填充可能掩盖信号 |
| ST 过滤 | high | 基本正确。但对散户策略影响程度因人而异，medium 也可 |
| 成分股历史时点 | medium | 偏低。幸存者偏差是回测失效的头号原因，应升为 high |
| 新股/次新股 | medium | 正确。注册制后影响有所降低，medium 合适 |

**建议**：将"成分股历史时点"的 severity 从 medium 升为 high，因为幸存者偏差是导致量化策略回测失真最常见、最系统性的原因。

### 遗漏的重要 business decision

以下决策在 `design_decisions` 或 `pseudocode_example` 中有体现，但未进入 business_decisions：

1. **全局 MultiIndex [entity_id, timestamp]**（data_reader 阶段）：这是 B 类决策。选择这个索引结构意味着所有因子计算必须符合这个约定，对 AI agent 编写自定义因子时有约束。

2. **AutoCode/Generator 自动生成 kdata Schema**（schema_layer 阶段）：这是 T 类决策，但值得标注，因为 AI agent 扩展新标的时需要知道"不需要手写 Schema"。

3. **filter_result NaN 在 AND 合并时的传播行为**：已标注为 B 类，但其 rationale 缺少具体示例——NaN AND True = NaN 还是 False？这直接影响多因子策略的合并逻辑。

## 2. A 股领域准确性

### 印花税率（0.05%）

**结论：正确。** 2023-08-28 起降至 0.05%（单边卖出）。

但有一处内部矛盾需要注意：

- `印花税含于 sell_cost` 条目的 rationale 写道：`sell_cost=0.001（0.1%）包含了该税项但使用了2023年前旧费率，对机构账户偏高`
- 而 `买入手续费默认 0.1%` 条目写道：`buy_cost=0.001（0.1%）对应A股中等水平券商佣金`

矛盾点：sell_cost 被批评为"旧费率偏高"，但 buy_cost 同等数值 0.1% 却被描述为"中等水平"，这两个判断逻辑不一致。实际上两个值都偏高（只是原因不同：sell 是因为印花税率降了，buy 是因为现在头部券商万分之三就够了）。

**建议修正**：buy_cost 的 rationale 也应补充"现在主流券商佣金已低至 0.003%~0.005%，0.1% 为偏保守估计，更适合散户账户"。

### 成本参数合理性

从实战角度：
- **buy_cost=0.001**：偏高。机构通常 0.003%，哪怕中小散户现在也在 0.02%~0.05% 之间。0.1% 已是过时的高佣金账户。
- **sell_cost=0.001**：2023 年后应为印花税 0.05% + 佣金，综合约 0.065%~0.12%。0.1% 是合理区间的上沿。
- **slippage=0.001**：对日线级别小市值股票偏低，对大市值蓝筹偏高。固定滑点模型的局限性已在 rationale 中指出，这是诚实的。

### -30% 止损阈值描述

**"教学/入门框架的设计取向"** 这个定性是准确的，且有说服力。行业 CTA 常用 -3%~-5%，公募量化约 -8%~-15%，-30% 确实是"几乎不止损"的教学模式。这个背景说明对 AI agent 非常重要——防止 agent 把 -30% 误解为"最佳实践"。

### severity 评级总结

除成分股历史时点应升为 high 外，其余评级合理。

### 缺失的重要用例

1. **多因子 AND/OR 组合策略**：TargetSelector 的 condition_and/condition_or 两种模式是框架核心能力，但 9 条用例中没有一个专门展示如何组合多个 Factor 的完整流程。MACD 示例只展示了单因子，多因子组合是实战的主要用法。

2. **回测结果分析与归因**：框架内置了 AccountStats 和净值曲线，但没有一个用例展示如何从回测结果中提取 Sharpe/MaxDrawdown 等指标，这是量化策略评估的基本需求。

3. **增量数据更新工作流**：框架支持增量更新（判断已有数据范围，只采集缺失部分），这是生产环境的核心场景，但没有对应用例。

### 用例描述质量

business_problem 字段整体写得好，以用户意图为中心（"如何做 X"）而非框架结构（"这是 Y 组件"），对 AI agent 的意图匹配有帮助。

唯一的问题：`均线多因子选股` 的 source 写 `examples/factors/`（一个目录而非具体文件），降低了可追溯性。

## 4. 实用价值

### 升级前 vs 升级后：AI agent 能多做什么

**升级前（纯架构蓝图）**：

AI agent 可以回答"zvt 的 Trader 是怎么工作的"、"怎么扩展数据源"等结构性问题，但无法区分：
- 哪些设计可以安全修改（不影响交易结果）
- 哪些修改会改变策略行为（需要业务层面决策）
- 哪些约束是法规强制（不能绕过）

**升级后（架构 + business_decisions + known_use_cases）**：

1. **防误修改**：当用户说"帮我把止损改为 -10%"，agent 可以结合 BA 标注知道：这不是 bug 修复，而是业务假设的替换，需要用户确认策略风格。

2. **合规意识**：当用户说"帮我写一个能在涨停当天买入的策略"，agent 可以从 RC 标注的涨跌停 known_gap 中感知到这是未处理的监管约束，主动告知实盘风险。

3. **场景匹配**：known_use_cases 的 business_problem 字段让 agent 可以做意图匹配（"用户想做基本面选股" → 推荐 fundamental_selector 用例），而不是从源码目录中盲目搜索。

4. **成本参数校正**：当 agent 帮用户生成回测代码时，可以主动提示"当前默认 sell_cost=0.001 使用了2023年前旧费率，建议更新为 0.00065"。

### T/B/BA/DK/RC 框架的价值评估

这个五分类框架的核心价值在于：**它强迫知识编码者区分"能改什么"和"不能改什么"**。

对 AI agent 而言，这个区分至关重要：
- T 类：可以安全替换（换数据库不影响策略）
- RC 类：不能绕过（违规）
- BA 类：可以替换但必须告知含义变化
- DK 类：需要领域知识才能安全替换
- B 类：用户的主动选择，agent 不应默认替换

这不是过度工程。对量化领域尤其重要，因为回测结果的正确性高度依赖这些隐含假设是否被正确理解。

## 5. 仍然缺失

### 已标注的 6 个 known_gap 之外，还缺少什么

**市场状态相关（缺失较多）**

1. **牛熊市制度切换**：zvt 的 Trader 没有市场状态感知，无法在系统性下跌时自动降仓。这是一个大 BA 类缺失——框架隐含假设市场状态稳定。

2. **流动性约束**：固定滑点 0.1% 是针对中等流动性股票的假设，对小市值股票（日均成交额 < 5000 万）严重低估冲击成本。蓝图未标注"策略容量"概念。

**A 股特有的数据质量问题**

3. **A 股数据源不一致性**：不同数据源（东方财富 vs 聚宽 vs 新浪）对同一只股票的复权系数可能不一致，导致跨数据源回测结果难以复现。这是使用 zvt 多数据源时的核心坑点，但蓝图未提及。

4. **财报数据滞后性**：zvt 的基本面数据（PE/PB/ROE）是 point-in-time 还是事后调整值（restatement）？这直接影响因子回测的前视偏差。蓝图未标注。

**季节性与日历效应**

5. **春节/财报季效应**：A 股在春节前后、年报/中报披露窗口有显著的异常收益模式，但蓝图完全没有 DK 条目覆盖这类市场日历效应。

**跨策略交互**

6. **因子相关性**：当用户组合 MACD 因子 + MA 因子 + 换手率因子时，三个因子之间的相关性（同为动量因子）会导致组合效果远差于预期，但蓝图没有任何关于因子相关性的 DK 条目。

**框架架构层面**

7. **并发安全**：zvt 的多 Recorder 同时运行是否有并发锁机制？当多个策略同时写入同一 SQLite 时的行为未记录。对生产环境部署的 AI agent 是重要信息。

8. **内存管理**：Accumulator 有状态累积在长期回测中的内存占用问题未标注（回测 10 年 500 只股票的内存峰值可能达到数 GB）。

## Top 3 优点

1. **RC 分类的引入是最大亮点**：将"T+1 信号延迟"、"印花税"、"涨跌停"从普通业务决策中独立出来，明确标为不可绕过的监管约束，这对防止 AI agent 生成违规策略有直接价值。

2. **rationale 的信息密度高**：每条 rationale 不只说"是什么"，还说"为什么这样而非那样"（如止损 -30% 对比了行业 CTA 的 -3%~-5%），这正是 AI agent 需要的决策上下文。

3. **known_gap 的 severity + impact 标注**：将缺失内容分级并说明影响（如涨跌停导致"动量策略回测收益率会系统性高估"），让 AI agent 能够根据用户策略类型判断哪些 gap 对当前场景是致命的。

---

## Top 3 不足

1. **复合类型（B/BA、T/B 等）缺少主从说明**：有 6 条使用了复合类型标注，但 YAML 结构中没有 `primary_type` 字段。AI agent 在需要"这是什么类型"的判断时会面临歧义。建议增加 `primary_type` 字段或在 rationale 中明确主分类。

2. **sell_cost 的内部矛盾**：rationale 批评 sell_cost=0.001 使用旧费率，但 buy_cost=0.001 同等数值却被描述为"中等水平"，逻辑不自洽。这会让 AI agent 生成的成本参数建议前后矛盾。

3. **市场结构与容量类 DK 完全缺失**：9 条 known_use_cases 和 23 条 business_decisions 都聚焦在框架的架构层，对"这个策略能承载多大资金规模"、"在什么市场环境下会失效"等容量/适用范围类知识是空白。这类知识对实盘落地是关键门槛。

## 结论：是否值得推广到其他 58 个蓝图

**条件性推广（Conditional Yes）**

推广前建议完成以下修正：

**必须修复（阻塞推广）**

- [ ] 统一复合类型的处理方式：增加 `primary_type` 字段，或规定复合类型的写法（"B/BA" 中第一个为主分类）
- [ ] 修复 sell_cost vs buy_cost rationale 的逻辑矛盾
- [ ] 明确 `"B/BA"` 等复合类型的语义：是"既是B又是BA"还是"主B次BA"？

**建议修复（提升质量）**

- [ ] 将成分股历史时点的 severity 从 medium 升为 high
- [ ] 除权除息分类补充 RC 分量
- [ ] 在 known_use_cases 中增加"多因子组合"和"回测结果分析"两个核心用例

**推广理由**：

T/B/BA/DK/RC 五分类框架本身设计合理，rationale 的写作规范明确，known_use_cases 的 business_problem 导向写法对 AI agent 友好。这套方法论经过四方评审校正，正确率高于单一 AI 审计。

修复上述矛盾后，该框架适合作为 SOP v2.3 的标准模板推广到金融领域的其他蓝图（如 Backtrader、VeighNa、AKShare 等），以及其他领域中同样存在"监管约束+业务假设混杂"问题的蓝图（如医疗、法律合规类框架）。
