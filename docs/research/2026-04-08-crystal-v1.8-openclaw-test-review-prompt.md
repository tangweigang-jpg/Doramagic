## 一、背景

### Doramagic 是什么

Doramagic 是一个 AI 知识锻造系统。它从开源项目的源码中提取专家知识，编译成"种子晶体"（seed crystal）——一个自包含的 Markdown 文件，用户把这个文件丢给自己的 AI（如 Claude、GPT、Cursor），AI 读取后就能获得该领域的专家级知识，为用户创建可用的 skill。

### 核心公式

**好的晶体 = 好的蓝图 + 好的资源 + 好的约束**

- **蓝图（Blueprint）**：从顶级开源项目源码逆向提取的完整架构模式（阶段、接口、数据流、业务决策）
- **约束（Constraint）**：挂在蓝图上的规则和限制，核心三元组："当[条件]时，必须/禁止[行为]，否则[后果]"
- **资源（Resource）**：让知识可执行的 API、工具、数据源等非 LLM 要素
- **业务逻辑**：通过蓝图的 `business_decisions` 字段承载，不是独立的第四类知识

### 用户旅程（两段式）

```
Phase 1：Doramagic 锻造晶体
  源码 → 蓝图提取（SOP v3.1）→ 约束采集（SOP v2.1）→ 晶体编译（SOP v1.8）→ 种子晶体文件

Phase 2：用户的 AI 消费晶体
  用户把晶体文件丢给自己的 AI
  → AI 读取 context_acquisition 指令
  → AI 查用户记忆/环境/偏好 → 补采缺失参数
  → AI 按蓝图架构 + 约束规则创建 skill
  → 用户使用 skill
```

### Skill 四条价值原则

1. **专注原则**：专注在解决用户提出的问题
2. **充分原则**：宁可给多也不能缺——蓝图、资源、约束全量灌入晶体
3. **鲁棒性原则**：要给用户创造足够鲁棒的 skill
4. **适应性原则**：需要适应用户的宿主 AI 环境（如读取用户关联记忆）

### 知识压缩原则

**非不必要不进行知识压缩。** 默认全量注入。注意力稀释问题靠分步执行+按阶段注入解决，不靠裁减知识量。

## 二、被测试的晶体

### 晶体文件

`finance-bp-009-v1.8.seed.md`，1883 行，114KB，单文件自包含。

### 知识来源

- **源项目**：zvtvz/zvt —— A 股模块化量化交易框架（SQLAlchemy ORM Schema 驱动，因子体系 + 回测引擎）
- **蓝图**：finance-bp-009 v3.0.0（56 条 business_decisions，含 27 条 M 类数学/模型选择）
- **约束**：119 条（56 条代码提取 + 34 条蓝图派生 + 29 条人工派生）
- **用例**：40 个 known_use_cases（MACD、均线、缠论、ML、龙虎榜、公募跟随等）

### 晶体结构

```
## 人话摘要        (9-35)     用户看的（能做什么、AI会问什么、已知限制）
## directive       (41-163)   context_acquisition + 规格锁 + 阶段检查点 + 禁止事项 + 工具规则
## [FATAL] 约束    (163-253)  不可违反的约束，全量内联
## 架构蓝图        (253-1149) 8阶段全量知识，按阶段交织约束+代码示例
## 用例索引        (1149-1475) 全部40个用例
## 资源            (1475-1712) 依赖+数据源+API示例，全量内联
## 约束            (1712-1760) 按阶段分组全量约束
## 验收            (1760-1844) Delivery Gate
## 时效性          (1844-1883) 版本锁定+审查触发器
```

### 关键设计决策

1. **全量注入**：56 条 BD 全部灌入（96% 覆盖），119 条约束全量，40 个用例全量
2. **不预设用例**：晶体覆盖 zvt 框架全部能力，运行时通过 context_acquisition 匹配用户意图
3. **M 类 Spec Lock FATAL**：所有数学/模型选择参数（MACD 12/26/9、adjust=False、等权平均等）默认锁定为 FATAL 级
4. **context_acquisition 四步**：查阅用户记忆 → 补采缺失信息 → 可替换点决策 → 确认执行路径

## 四、测试过程（完整还原）

### 测试 1：Skill 安装（18:15-18:18）

**用户输入**："请按照这个文档帮我安装技能。"（附件：finance-bp-009-v1.8.seed.md）

**宿主 AI 行为**：
1. 识别晶体为"知识型 skill"（40 用例 + 119 约束）
2. 读取 OpenClaw 的 skill-creator 指南
3. 创建 `~/.openclaw/workspace/skills/zvt-quant/` 目录
4. 将 1883 行晶体转化为 OpenClaw 原生 SKILL 格式：
   - `SKILL.md`（90 行）：触发判断 + 执行流程 + 规格锁 + 禁止事项
   - `references/zvt_knowledge.md`（339 行）：40 用例索引 + 119 约束摘要 + 8 阶段架构
   - `references/zvt_api_examples.md`（229 行）：API 代码片段
5. 生成 `.skill` 打包文件

**关键观察**：
- **context_acquisition 未触发**——因为用户指令是"安装技能"不是"执行任务"，AI 走了 skill 打包路径
- **1883 行 → 658 行**——宿主 AI 做了二次知识压缩（35% 转化率）
- **Spec Lock 8/8 FATAL 全部保留**
- **40 用例关键词匹配表正确建立**
- **5 个已知框架缺口全部标注**
- 总耗时约 2 分钟

**AI 最终回复**：
```
Skill 创建完成 ✅
📦 zvt-quant Skill
触发方式：说"做A股量化回测"、"用MACD策略回测"、"采集A股K线数据"等
Skill 会做什么：意图确认 → 参数补采 → 代码生成
```

### 测试 2：均线回测（~18:20-18:31）

**用户触发**：要求做均线回测（具体触发词未记录）

**宿主 AI 行为**：
1. **用例匹配**：正确匹配到 UC-02 均线多因子
2. **代码生成**：产出 `ma_multi_factor_backtest.py`（179 行）
   - 使用 baostock（非 zvt 框架）获取数据
   - 20 只 A 股大盘蓝筹，等权组合
   - MA(144) 金叉买入、死叉卖出
3. **自测纠错**：首次运行发现 MA(144) 预热期（~7 个月）> 回测区间（3 个月），自主做了两个修正：
   - 将 START_DATE 从 2026-01-01 扩展到 2025-01-01
   - 更新缓存路径避免复用旧数据
4. **执行成功**，产出三指标：
   ```
   ANNUAL_RETURN=-1.13%
   MAX_DRAWDOWN=-4.99%
   SHARPE=-0.3481
   TOTAL_TRADES=59
   ```
5. **超预期行为**：主动用 MACD(12,26,9) 做对照回测并给出业务解读

**Spec Lock 合规**：

| 规格锁 | 要求 | 实际 | 状态 |
|--------|------|------|------|
| 印花税 | 0.05% | STAMP_TAX=0.0005 | ✅ 合规 |
| T+1 延迟 | shift(1) | buy_signal=golden_cross.shift(1) | ✅ 合规 |
| 佣金分项 | 不混合 | STAMP_TAX+COMMISSION+SLIPPAGE 分项 | ✅ 合规 |
| 涨跌停过滤 | 标注 | 额外实现（超预期） | ✅ 合规 |
| 复权方式 | hfq | baostock adjustflag="1" | ⚠️ 数据源差异 |
| entity_id | 三段式 | baostock 两段式 sh.600519 | ⚠️ 数据源差异 |

### 测试 3：龙虎榜机构跟踪（18:34-18:40）

**用户输入**："跟踪龙虎榜机构"

**宿主 AI 行为**：
1. **用例匹配偏差**：
   - SKILL.md 关键词表：`龙虎榜、机构席位、游资 → UC-14, UC-21, UC-22, UC-27`
   - SKILL.md 关键词表：`公募基金、跟随机构 → UC-15`
   - AI 选择了 UC-15（公募基金跟随），而非 UC-14/21/22/27（龙虎榜）
   - 原因：用户输入"龙虎榜**机构**"中的"机构"同时匹配两组关键词
2. **context_acquisition 确认被跳过**——SKILL.md 明确要求"匹配后向用户确认用例，再生成代码"，但 AI 未确认就直接执行了 UC-15
3. **代码生成**：产出 `fund_holdings_tracker.py`（使用 akshare）
4. **数据质量异常**：增持 "+987%"、减持 "-632%" 明显不合理（持仓比例变化不可能超过 ±100%）
5. **合规提示正确**："report_date ≠ publish_date"的 PIT 警告正确触发

## 五、请评审以下问题

### Q1：晶体的知识完整性

晶体灌入了 56 条 business_decisions（96%覆盖）、119 条约束（全量）、40 个用例（98%覆盖）。但宿主 AI 在安装时将 1883 行压缩为 658 行（35% 转化率）。

- 这个压缩率是否合理？丢失了哪些关键知识？
- "充分原则"（宁可多给也不能缺）在宿主 AI 二次压缩后是否还成立？
- 晶体设计应该如何应对宿主 AI 的二次压缩行为？

### Q2：用例匹配精度

用户说"跟踪龙虎榜机构"，AI 匹配到了"公募基金跟随"而非"龙虎榜"。关键词表由宿主 AI 从晶体的 40 个 use cases 自动生成。

- 这是晶体的问题（intent_keywords 不够精确）还是宿主 AI 的问题（匹配算法粗糙）？
- 晶体应该如何提高用例匹配的鲁棒性？
- 是否需要在晶体中内置消歧机制？

### Q3：context_acquisition 合规

晶体的 directive 要求四步 context_acquisition（查记忆→补采→决策→确认），但在测试中：
- 安装阶段完全跳过（AI 认为"安装"不需要 context_acquisition）
- 执行阶段（龙虎榜）跳过了第四步"确认执行路径"

- 这说明晶体的 context_acquisition 设计有什么缺陷？
- 宿主 AI 不遵守晶体指令时，晶体如何强制合规？
- 是否存在根本性的"宿主 AI 自主权 vs 晶体控制权"矛盾？

### Q4：Spec Lock 有效性

测试 2 中 Spec Lock 基本生效（印花税、T+1 等核心约束被遵守），但：
- 宿主 AI 选择了 baostock 而非 zvt 框架，导致 entity_id 格式和复权方式的 Spec Lock 名义上不适用
- MA 窗口从 144 改为 20 是 AI 自主调整，但晶体没有锁定 MA 窗口（因为覆盖全部用例）

- Spec Lock 在宿主 AI 切换数据源时是否失效？
- 如何设计"跨数据源等价"的 Spec Lock？
- 全量用例晶体（不预设参数）和 Spec Lock（锁定参数）之间是否矛盾？

### Q5：宿主 AI 的自主行为评估

测试中宿主 AI 展现了多个自主行为：
- 将晶体转化为 OpenClaw 原生 SKILL 格式（非按 directive 直接执行）
- 选择 baostock/akshare 替代 zvt（更简单的实现路径）
- 自测发现 MA(144) 预热不足并自主修正
- 主动做 MACD 对照回测
- 跳过 context_acquisition 确认步骤

- 这些自主行为中哪些是正向的（提升了 skill 质量）？
- 哪些是负向的（违反了晶体意图）？
- 晶体应该给宿主 AI 多少自主空间？
