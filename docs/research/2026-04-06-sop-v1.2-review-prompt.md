## 一、产品背景

### Doramagic 是什么

Doramagic 是一个**知识提取与封装系统**。它从 GitHub 开源项目中提取领域知识，将其封装为可部署的"知识晶体（Crystal）"，让 AI agent 在各种宿主环境中执行有根基的专业任务。

核心价值主张：**AI 执行任务时，有领域知识注入 vs. 没有领域知识注入，产出质量存在本质差异。** Doramagic 的晶体就是这个知识注入层。

### 关键认知：不只是金融

**Doramagic 的最终产品面向所有行业**。当前实验平台选择了金融量化，是因为：
- 有明确的数值化验收标准（年化收益率、夏普比率等）
- 失败模式清晰、可复现
- 适合快速验证 SOP 正确性

但晶体的目标场景涵盖（但不限于）：
- **Web 开发**：从 Next.js / Django / FastAPI 等框架提取最佳实践，生成符合框架惯例的应用骨架
- **数据工程**：从 dbt / Airflow / Spark 等工具提取 pipeline 设计模式
- **AI/ML 工程**：从 LangChain / LlamaIndex / vLLM 等框架提取集成模式
- **移动端开发**：从 React Native / Flutter 等提取组件设计模式
- **DevOps/平台工程**：从 Kubernetes / Terraform 等提取基础设施配置模式

**请在评审时始终牢记这一跨行业目标。** 一个 SOP 若只能在金融量化场景下工作，则设计是失败的。

### 执行宿主：OpenClaw

当前主要宿主是 OpenClaw——一个基于 LLM 的 agent 执行平台，通过 Telegram 接收指令并执行。关键参数（来自配置文件实测）：

| 参数 | 值 | 来源 |
|------|---|------|
| 超时 | 1800 秒（30 分钟） | `openclaw.json` |
| 主模型 | GLM-5，Context 204K | `openclaw.json` |
| exec 执行 | 可用，但拦截链式 shell 命令（`&&`, `;`, `\|`） | v5-v9 实测 |
| Skill 加载 | 读取 SKILL.md + references/ 目录 | 平台规范 |
| `{baseDir}` 占位符 | 在 SKILL.md body 中**不插值** | v13 实测 |

### 晶体编译 SOP v1.2

> **定位**：此 SOP 是蓝图提取 SOP v2.3 和约束采集 SOP v1.3 的下游消费者。
> 输入是已提取的蓝图+约束+资源，输出是可部署的晶体。

**核心原则**

晶体 = 好的蓝图 + 好的资源 + 好的约束 + 好的 Harness

四层缺一不可。Harness 是让知识正确发挥作用的手段，不能替代知识本身。

晶体的核心价值是让 AI 获得它原本没有的领域知识。如果晶体只有 Harness 没有知识，那它和一份写得好的 prompt 没有本质区别。

#### 输入

| 输入 | 来源 | 要求 |
|------|------|------|
| Blueprint YAML | 蓝图提取 SOP v2.3 产出 | 含 business_decisions（T/B/BA/DK/RC 标注）+ known_use_cases |
| Constraints JSON/JSONL | 约束采集 SOP v1.3 产出 | 含 severity、constraint_kind、stage_ids |
| Resources | 蓝图 YAML 的 replaceable_points | 含 options、fit_for、not_fit_for |
| User Intent | 用户或产品经理提供 | 目标市场、策略类型、产出类型 |
| Target Host | 部署决策 | openclaw / claude_code / generic |
| Host Spec | `docs/research/host-specs/{host}-host-spec.md` | 宿主能力规范，不编撰 |

#### 输出

| 输出 | 格式 | 用途 |
|------|------|------|
| Crystal IR | `crystal-{id}.ir.yaml` | 结构化中间表示，所有知识+harness 的唯一真相源 |
| Rendered seed.md | 每个目标宿主一份 | 交给 AI agent 执行的最终文件 |

#### Step 1: Execution Contract 编写

**1a. Spec Lock 提取**

输入：蓝图 `business_decisions` 中 type=B 或 type=BA 的条目。

规则：
1. 扫描所有 B/BA 条目，判断哪些参数在 AI 执行时会漂移
2. 判断标准（来自 v5-v9 教训）：
   - 数值参数（TOP_N、窗口期、股票池大小、成本率）→ 必须锁
   - 策略类型/方法选择 → 必须锁
   - 数据源选择 → 建议锁
   - 输出路径/格式 → 必须锁（使用 `{workspace}` 占位符）
   - 市场/品种选择 → 必须锁
3. 每条 spec_lock 格式：
   ```yaml
   - param: {参数名}
     value: {锁定值}
     violation: FATAL | WARN
     rationale: {为什么锁这个值}
   ```
4. type=RC 的条目（监管规则）直接标为 `violation: FATAL`，不可协商

质量标准：spec_lock 覆盖所有 type=B/BA 的可漂移参数。

**1b. Delivery Gate 提取**

输入：`known_use_cases[target].must_validate` + 约束中 severity=fatal 的条目。

规则：
1. 每条 delivery_gate 必须是可程序化验证的
   - 好的：`abs(annual_return) > 0.01`
   - 坏的：`结果看起来合理`
2. 必须包含业务合理性检查：
   - `abs(annual_return) > 0.01`（v5 教训：-0.25% 是 bug 伪结果）
   - `trade_count > 0`（零成交 = 信号逻辑 bug）
3. 必须包含规格一致性检查：
   - 策略类型未漂移
   - 关键业务规则已应用（如 T+1 延迟）
4. 每条格式：
   ```yaml
   - id: DG-{nn}
     condition: {可验证的条件}
     on_fail: RERUN | BUG_INVESTIGATION | REBUILD
   ```

质量标准：delivery_gate 至少 5 条。覆盖文件存在性、指标完整性、业务合理性、规格一致性四个维度。

#### Step 2: Failure Taxonomy 编写

**2a. 通用失败类加载**

以下 6 种失败类适用于所有晶体（从 v1-v9 全部失败案例抽象）：

| 失败类 | 触发条件 | 恢复动作 | 禁止动作 |
|--------|---------|---------|---------|
| `exec_rejected` | 宿主拒绝执行命令 | 重写命令为白名单格式 | 不重试相同命令 |
| `param_drift` | spec_lock 参数偏离锁定值 | 立即恢复锁定值 | 不辩解修改理由 |
| `timeout_risk` | 预估剩余工作 > 50% 时间预算 | 简化方案 | 不增加工具调用 |
| `framework_dead_end` | 框架 API 错误重试 2 次仍失败 | 切换替代方案 | 不继续重试 |
| `mode_switch` | 产出文档而非代码 | 删除文档，重新生成代码 | 不产出参考文档 |
| `silent_drift` | 策略偏离规格但未报错 | 对照 spec_lock 逐项检查 | 不假设"接近"可以 |

**2b. 领域特定失败类派生**

输入：蓝图 `business_decisions` 中 `status: missing` 且 `known_gap: true` 的条目 + 约束中 severity=fatal/critical 的条目。

规则：
1. 只有 severity=critical 或 severity=high + known_gap=true 的条目才派生
2. 每种失败类必须有具体 recovery action（不能只说"注意"）
3. 格式：
   ```yaml
   - name: {失败类名}
     trigger: {触发条件}
     recovery: {具体恢复动作}
     source: {来源蓝图条目或约束 ID}
   ```

质量标准：领域特定失败类至少 2 条，来自不同的蓝图阶段。

#### Step 3: Stage Spec 编写

**3a. 阶段拓扑设计**

默认四阶段：

```
setup → execute → validate → deliver
```

可根据蓝图 stages 扩展，但最多 4-5 个阶段。超过 5 个阶段会增加 AI 的认知负荷。

规则：
1. 每个阶段必须定义：
   - `goal`：一句话描述目标
   - `actions`：具体动作列表
   - `exit_criterion`：退出条件（可程序化验证）
   - `budget_ratio`：时间预算占比（基于宿主实际 timeout）
2. 所有阶段 budget_ratio 之和 = 1.0
3. 禁止"可选扩展阶段"（v8 教训：AI 会直接跳到扩展）
4. 禁止跳阶段：必须从 1 到 N 顺序执行
5. 时间预算必须从 Host Spec 读取，不编撰

**3b. 约束分配**

每个阶段声明 `constraints_to_load`，列出该阶段应注入的约束类别。

规则：
- severity=fatal 的约束所有阶段都注入
- 其他约束按 `stage_ids` 匹配注入
- 约束注入数量上限由渲染步骤根据宿主注意力特性决定，不在此步骤硬编码

#### Step 4: Host Adapter 编写

原则：
1. 白名单优先：先定义 AI 可以做什么，不列举不能做什么
2. 引用 Host Spec：所有宿主能力参数从 Host Spec 文档读取，不编撰
3. 路径占位符：使用 `{workspace}` 代替硬编码路径

格式：
```yaml
host_adapters:
  {host_name}:
    spec_ref: "docs/research/host-specs/{host}-host-spec.md"
    timeout_seconds: {从 Host Spec 读取}
    exec_rules:
      whitelist: [...]
    file_strategy: {根据 timeout 和任务复杂度决定}
    path_template: "{workspace}/{filename}"
```

#### Step 5: 知识压缩

问题：蓝图和约束不能全量灌入晶体，需要系统化的知识压缩方法。

**5a. 知识三维筛选**

每条知识按三个维度打分：

**维度 1：AI 新颖度**——AI 是否已经知道这个知识？

| 等级 | 判断标准 | 处理 |
|------|---------|------|
| HIGH | AI 大概率不知道的领域特有知识 | 必须包含，完整描述 |
| MEDIUM | AI 可能知道但容易出错的知识 | 包含，简要提醒 |
| LOW | AI 几乎肯定知道的通用知识 | 省略或一句话带过 |

**维度 2：失败预防度**——这条知识是否防止已知失败？

| 等级 | 判断标准 | 处理 |
|------|---------|------|
| CRITICAL | 直接防止某个具体历史失败 | 必须包含 |
| HIGH | 防止蓝图 known_gap 中标注的风险 | 应当包含 |
| LOW | 通用最佳实践，无对应失败案例 | 可省略 |

**维度 3：阶段相关度**——这条知识在哪个执行阶段需要？

| 等级 | 处理 |
|------|------|
| 当前阶段直接相关 | 注入当前阶段 |
| 跨阶段通用 | 注入 Contract 或全局知识节 |
| 其他阶段专用 | 放入 references/ 按需加载 |

**5b. 压缩规则**

1. 必须包含（HIGH 新颖度 OR CRITICAL 失败预防）：完整描述，不压缩
2. 简要包含（MEDIUM 新颖度 AND HIGH 失败预防）：一句话 + 关键数值
3. 按需引用（仅在特定阶段需要）：放入 references/ 目录，声明加载触发条件
4. 省略（LOW 新颖度 AND LOW 失败预防）：不注入晶体

**5c. 质量标准**

- 蓝图 business_decisions 覆盖率：type=RC 100%，type=B ≥ 80%，type=BA/DK 按新颖度筛选
- 约束覆盖率：severity=fatal 100%，severity=high + 阶段匹配 ≥ 60%
- 资源覆盖率：本用例直接使用的资源 100%

#### Step 6: Crystal IR 组装

将 Step 1-5 产出组装为结构化 YAML，分为**知识层 + 执行层**两层：

```yaml
version: "1.2"
crystal_id: "finance-bp-{id}-{use_case}"
references:
  blueprint: "path/to/blueprint.yaml"
  constraints: ["path/to/constraints.jsonl"]
  use_case: "use_case_name"

# 知识层（蓝图+约束+资源，经 Step 5 压缩）
knowledge:
  business_decisions: [...]
  constraints:
    fatal: [...]
    high_relevant: [...]
  resources:
    data_sources: [...]
    dependencies: [...]
    storage: [...]
  references:
    - path: "references/blueprint-full.md"
      content: "完整蓝图知识"
      load_when: "AI 需要理解框架完整架构时"

# 执行层（Harness）
harness:
  contract: { ... }
  failure_taxonomy: { ... }
  stage_spec: { ... }
  host_adapters: { ... }

compilation:
  sop_version: "crystal-compilation-v1.2"
  compiled_at: "ISO date"
```

#### Step 7: 宿主渲染

**渲染原则**：
1. 知识与 Harness 平衡：领域知识和 Harness 各占约 50%
2. 路径参数化：`{workspace}` 占位符，由编译器在渲染时替换
3. 渲染规则从 Host Spec 派生，不编撰
4. Progressive Disclosure：核心知识内联，深度知识放 references/ 按需加载
5. 三种渲染目标：seed.md（一次性执行）、SKILL.md（可复用 skill）、bundle/（完整 skill 包）

**7a. seed.md 渲染**

适用场景：直接发给 AI 执行的消息（如通过 Telegram 发给 OpenClaw）。

{exec_whitelist}

3. 读取 `scaffolds/{output-file}.scaffold.md` 获取输出骨架

SKILL.md 渲染规则：
- SKILL.md 正文 ≤ 80 行（经验值，非硬限制）
- 重知识放 references/（利用宿主按需加载）
- 输出格式控制用 scaffold + REPLACE_WITH（不让 AI 自由发挥）
- 不依赖 exec（普通用户未配置 exec-approvals）

**7c. {workspace} 占位符解析**

| 渲染目标 | 谁解析 | 解析为 |
|---------|--------|--------|

| scaffold | AI（填充骨架时） | 用户确认的工作目录 |

| references 完整 | 蓝图全量、约束全量、API 示例均在 references/ |

## 三、演进历史与设计教训

理解这段演进历史，有助于评审者明白为什么 SOP 做出了某些看似繁琐的设计决策。

### v1.0：首次实测，发现认知错误

**设计**：7 步流程，重点在 Harness（Execution Contract + Failure Taxonomy + Stage Spec）。知识注入作为辅助考虑。

**测试结果**：
- 蓝图覆盖率仅 **3%**（蓝图 1249 行，晶体只注入了寥寥数条）
- 约束覆盖率仅 **6%**（133 条约束，仅 8 条进入晶体）
- 平台参数被编撰（超时写成 600s，实际是 1800s；虚构了 8KB 体积限制）
- 产出了 seed.md，但没有产出可复用 skill（SKILL.md）

**核心错误**：把 Harness 当成了晶体的主体，把知识当成了附属品。

### v1.1：修正事实错误

**改动**：
- 修正 timeout 600s → 1800s（来自 `openclaw.json` 实测）
- 移除虚构的 8KB 体积限制和 `max_tool_calls: 5`
- Host Adapter 改为引用 Host Spec，不内联数值
- 路径使用 `{workspace}` 占位符
- 新增知识密度要求（文字表述，无量化标准）

**遗留问题**：知识压缩方法仍然粗糙；SKILL.md 渲染路径缺失。

### v1.2：当前版本

**主要改动**（相比 v1.1）：
1. **新增 Step 5：知识压缩**——三维筛选框架（新颖度 × 失败预防 × 阶段相关度）
2. **新增 SKILL.md 渲染路径**——完整的 SKILL.md 格式、目录结构、scaffold 机制
3. **量化覆盖率标准**——RC 100%, B ≥ 80%, fatal 100%, high 相关 ≥ 60%

### 尚未解决的争议

以下是我们内部尚有争议的点，欢迎评审者提供外部视角：

1. **"指令诅咒" 0.95^N 公式**：最初用这个公式解释为什么约束不能全量注入（每条约束有 5% 遗忘概率）。但这个公式未经实测验证，已在教训日志中标注（L11）。v1 实测的 35% 约束覆盖率，可能来自蓝图拆散混编的结构问题，而非纯粹的数量问题。
2. **80 行 SKILL.md 上限**：这是经验值。已有 1049 行的 skill 在 OpenClaw 上正常运行，80 行可能过于保守。
3. **"约束精选"的因果机制**：目前假设按阶段精选约束优于全量注入，但这个假设的因果机制（是约束太多导致注意力稀释？还是约束混在一起失去结构？）仍不清楚。

### Q2：Stage 结构的适用边界

**`setup → execute → validate → deliver` 这个四阶段结构是否足够通用？**

金融量化场景下，这四阶段对应：环境准备 → 运行回测脚本 → 检查指标 → 保存报告。

但考虑以下不同类型的"构建任务"：
- **搭建 Web 应用**（多文件、多服务、需要迭代调试）
- **配置 CI/CD Pipeline**（声明式，不是跑脚本）
- **训练 ML 模型**（长时间任务，中间有 checkpoint）
- **重构遗留代码**（没有明确的"产出文件"终点）

这个阶段结构是否足以覆盖上述场景？如果不足，你会如何扩展或改变这个结构，同时保持"最多 4-5 个阶段"的认知复杂度限制？

### Q3：知识压缩三维筛选

- 三维打分是否会产生大量"中间地带"（MEDIUM/HIGH/MEDIUM）的知识，导致判断困难？
- 请用一个**非金融领域**的例子（比如 Next.js 框架知识、dbt 最佳实践、K8s 配置模式），说明你会如何应用这个三维框架做筛选决策。

### Q4：SKILL.md 渲染路径的实用性

**v1.2 新增的 SKILL.md 渲染路径（SKILL.md + references/ + scaffolds/）是否足够实用？**

请评估：
- scaffold + REPLACE_WITH 占位符机制是否是控制 AI 输出格式的有效方式？有什么更好的替代方案？
- "SKILL.md 正文 ≤ 80 行，重知识放 references/"——这个分层策略的前提是 AI 会主动读取 references/ 文件。这个前提是否可靠？如何使其更可靠？

- 一个非技术用户通过自然语言触发 skill，和一个技术用户通过命令行触发 skill，这两种场景对 SKILL.md 设计的要求有什么不同？当前设计偏向哪种用户？

### Q5：Delivery Gate 的可迁移性

**"Delivery Gate 必须是可程序化验证的"这条规则在所有场景下是否成立？**

SOP 中的验收标准示例都是数值型的（`abs(annual_return) > 0.01`、`trade_count > 0`）。

但考虑：
- **Web 应用晶体**：验收标准可能是"页面在 localhost:3000 成功渲染"——这能程序化验证吗？
- **代码质量晶体**：验收标准可能是"代码通过 lint 和类型检查"——容易，但"代码符合架构规范"怎么验证？
- **文档生成晶体**：验收标准可能是"文档覆盖了所有公共 API"——如何程序化验证"覆盖了"？

当验收标准本质上是模糊的或主观的时，SOP 应该怎么处理？"可程序化验证"是否是一个过强的要求？

### Q6：晶体知识的时效性管理

**SOP 没有讨论晶体知识的时效性问题。这是一个盲点吗？**

知识晶体的核心价值是"注入 AI 原本没有的领域知识"。但：

SOP 是否需要一个"知识保鲜机制"（validity period、version pinning、deprecation 策略）？如果需要，它应该在哪个 Step 里体现？

**如果你是一个完全不了解 Doramagic 的工程师，需要使用这份 SOP 为一个新的 GitHub 项目（比如 FastAPI）编译第一个晶体，你会在哪个步骤卡住？**
