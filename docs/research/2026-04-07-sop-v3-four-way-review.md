# SOP v3.0 四方挑战评审综合

日期: 2026-04-07
评审方: Grok, Gemini, GPT（独立评审后综合）
评审对象: Finance SOP 从两级层级结构升级的方案

## 评审背景

当前 Finance SOP 的领域特定内容几乎只覆盖"量化交易"子领域（11 项 A 股必审清单），而 59 个蓝图中 76% 不是量化交易。提出的方案是将 SOP 改为"通用层 + Finance 通用层 + 子领域层"两级结构。

## 强共识（三方一致）

1. **树形结构不适用**：金融项目是多标签的（一个 repo 可能同时涉及定价+风险+回测），不能放进单一子领域。三方都建议改为可组合模块。

2. **13 个子领域分类轴混乱**：混合了业务域（定价）、技术范式（AI/LLM）、系统形态（数据基础设施）、市场结构（HFT）——不是同一分类轴。建议降维合并。

3. **T/B/BA/DK/RC 不够用**：对衍生品定价、风险建模等数学模型驱动的领域，缺少数学/统计模型分类。三方都建议增加 M 类。

4. **缺少横切关注点**：时间语义、数值精度、数据谱系、审计追踪、守恒约束——这些跨所有子领域，必须独立成模块。

5. **需要项目分类器**：AI agent 无法自行判断属于哪个子领域。需要显式的 Step 0 做多标签路由。

6. **SOP 应是编译产物**：三方都指出 SOP 不应是静态大文档，而应从规则模块按需编译。

## 关键分歧

| 维度 | Grok | Gemini | GPT |
|------|------|--------|-----|
| 合并粒度 | 7-8 宏子领域 | 4 大 Meta-Pillars | 多轴分类（业务域×系统形态×方法） |
| 业务逻辑分类 | 七类(+ML, SM) | 六类(+M) | 两层分类（逻辑性质×业务场景） |
| 执行方式 | Constraint Graph + Agentic Reviewer | 组件化拼接(orchestrator.py) | 规则包 + 激活器 + 编译器 |
| 落地节奏 | 2周全量迁移 | 立即重构 | 先3个模块压力测试，再扩展 |

## 实际采纳的决策

基于三方评审，SOP v3.0 最终采纳：
1. **分类升级**：T/B/BA/DK/RC → T/B/BA/DK/RC/M（六类，采纳 Gemini 方案）
2. **子领域合并**：13 → 7 个子领域（TRD/PRC/RSK/CRD/CMP/DAT/AIL）
3. **项目指纹**：新增 Step 0 项目指纹探针（三方共识）
4. **横切关注点**：20 项金融通用必审清单（三方共识）
5. **保持 SOP 为文档**：未采纳编译器方案（回到本质：SOP 的消费者是 AI agent + 提取管线是 Claude Code，不需要编译器）

## 未采纳但记录的建议

- Grok: Dynamic Constraint Graph (Neo4j) — 当前规模不需要
- Gemini: 状态机 SOP 调度器 (orchestrator.py) — 过度工程
- GPT: 规则包 + 激活器 + 编译器 — 方向正确但当前阶段不需要代码化
- GPT: 两层分类（逻辑性质×业务场景）— 设计文档中记录为 v2 架构方向，SOP v3.0 暂用单层六类
