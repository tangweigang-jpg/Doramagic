## 背景（给研究者的上下文）

我们正在构建一个叫 Doramagic 的产品——它从开源项目中提取专家知识，编译成"种子晶体"（一种 AI 可消费的知识配方），用户把晶体交给自己的 AI 工具，AI 按配方构建个性化 skill。

实验已验证：同一个 LLM，有种子晶体时在 8 项生产特性中命中 7 项，无种子晶体时命中 0 项。

我们已经做过一轮知识架构设计（三层模型：知识层 + 资源层 + 经验层），但在实践中发现了一个重大缺陷，现在需要验证新的架构方案。

## 提示词

```
我正在设计一个 AI 知识系统的架构。请你作为一个独立的架构评审者，评估我们的发现和新方案。

## 我们发现的问题

我们最初设计了一个三层知识模型来组织从开源项目中提取的知识：

- **知识层**：客观规则（"金融计算必须用 Decimal 不能用 float"）
- **资源层**：工具边界（"yfinance 数据延迟 15-20 分钟"）
- **经验层**：社区实践（"上线前必须 dry-run 72 小时"）

每条知识用统一的句式表达："当[条件]时，必须/禁止[行为]，否则[后果]"。

实践中我们发现了两个问题：

**问题 1：三层本质相同。**
追溯每条规则的来源，无论我们叫它"知识"还是"经验"，全都是"有人踩了坑，总结出的教训"。区别只是置信度和适用范围不同——"用 Decimal 不用 float"和"必须 dry-run 72 小时"的唯一区别是前者更普遍、后者更具体。它们的本质都是约束规则。

**问题 2：缺少架构/流程知识。**
我们能提取到大量的"规则"（do/don't），但提取不到"架构模式"和"业务流程"。比如：
- 规则："回测时必须使用复权价格" — 能提取 ✓
- 架构："量化系统应该分为 数据采集→信号生成→交易执行→风控 四个阶段" — 提取不到 ✗
- 流程："风控必须在执行之后、结算之前运行" — 提取不到 ✗

一颗只有规则没有架构的种子晶体，就像给装修工人施工规范但不给户型图——知道每条规范，但不知道房子长什么样。

## 我们的新方案

将知识体系从"三层平行"改为"两层嵌套"：

**蓝图层（Blueprint）**：架构结构——"房子长什么样"
- 来源：从开源项目的核心代码中提取架构骨架
- 形态：最小可运行的代码结构（类定义、函数签名、阶段划分、数据流向）
- 一个任务可以有多个蓝图（不同解法），一个蓝图来自一个或多个开源项目
- 这是用户和裸 AI 都想不出来的——核心价值所在

**约束层（Constraints）**：所有规则——"盖房子时不准犯的错"
- 将原来的知识/资源/经验三层合并，统一用句式"当[条件]时，必须/禁止[行为]，否则[后果]"
- 按置信度和适用范围分级（而不是按"层"分类）
- 约束附着在蓝图上——每条约束标注它适用于蓝图的哪个阶段/模块

最终产出（种子晶体）= 蓝图 + 附着的约束 + 验收标准 + context_acquisition 指令

## 你的任务

请用"帮我做一个股票投资分析工具"作为具体场景，回答以下问题：

### A. 评估新方案

1. "蓝图 + 约束"两层模型是否比"知识 + 资源 + 经验"三层模型更准确地反映了知识的本质？为什么？
2. 我们说"三层本质相同，都是踩坑后的规则"——这个判断是否正确？是否有反例？
3. 蓝图（架构结构）和约束（规则）之间的关系，是否还遗漏了第三种知识类型？
4. 新方案有什么潜在的盲点或风险？

### B. 蓝图的本质

5. 用股票投资工具举例：一个蓝图应该包含哪些元素？请给出具体的蓝图结构示例。
6. 蓝图和约束的边界在哪？什么知识属于蓝图，什么属于约束？给出 5 个边界模糊的例子和你的判断。
7. 一个好的蓝图和一个差的蓝图，区别在哪？如何评估蓝图的质量？
8. 蓝图如何从开源项目的代码中提取？提取的难点是什么？

### C. 挑战我们的假设

9. 有没有可能"蓝图"也是一种"约束"的特殊形式？（即两层模型也可以进一步合并为一层？）
10. 有没有可能我们仍然需要三层或更多层，只是分层维度错了？如果你来设计，你怎么分？
11. 同一个种子晶体在不同用户、不同 AI 工具中都要有效——这个跨平台需求对架构设计有什么约束？
12. 如果我们的方案有根本性错误，请直接指出。宁可被否定，不要被误导。

### D. 竞争对手视角

13. 如果你是 Cursor / GitHub Copilot / Devin 的架构师，看到 Doramagic 的"蓝图 + 约束"方案，你会怎么反击？
14. 这个方案的护城河在哪？什么是容易复制的，什么是难以复制的？

## 要求

- 不要泛泛而谈，用股票投资工具的具体例子填充每个观点
- 如果你认为我们的方案有根本性问题，请直接说，不要礼貌性赞同
- 区分"逻辑上正确但实践中困难"和"逻辑上就有问题"
- 请以 markdown 格式输出，保持结构清晰
```
