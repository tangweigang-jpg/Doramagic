# 三个关键决策点的跨模型挑战提示词

> 将以下三段提示词分别发给 GPT、Gemini、Grok。每个模型需要回答全部三个决策点。
> 保存为: schema-pipeline-challenge-gpt.md / gemini.md / grok.md

## 提示词

```
## 背景

我们正在为 Doramagic（一个AI顾问产品）设计知识采集流水线。

核心哲学是"用 LLM 约束 LLM"：
- 采集阶段：LLM 从 GitHub 项目、社区讨论、技术文档中提取结构化的"判断"（原子化知识单元）
- 消费阶段：这些判断被编译成"种子晶体"，注入用户的AI工具，约束 LLM 在具体场景下正确工作

判断的核心格式是：
> 当 [条件] 时，必须/禁止 [行为]，否则 [后果]

例子：
- 当进行金融计算时，禁止使用float，否则累计PnL偏差超0.01%
- 当需要实时交易决策时，禁止依赖yfinance，否则15分钟延迟导致决策失效
- 当上线前，必须先dry-run 72小时，否则首周出现致命bug

判断按三层专家本体分类：
- **知识层**：客观规律、因果关系（来源：跨项目代码共性 + 学术文献）
- **资源层**：工具/API的真实能力边界（来源：代码仓库 + API文档 + 实际测试）
- **经验层**：实践验证、失败模式、社区踩坑（来源：GitHub Issues/PR + 社区论坛）

我们已经设计了完整的五阶段流水线（定向→提取→清洗→入库→检索），现在需要你挑战以下三个关键设计决策。

---

## 决策点 1：经验层的社区知识过滤规则

### 当前方案
从 GitHub Issues 中提取经验层判断时，第一道过滤用确定性规则（不用LLM）：
- body_length >= 80 字符
- comments >= 3
- 包含技术关键词或代码块
- 排除纯 feature request / documentation / question 类型

过滤后再送给 LLM 做分类和判断提取。

### 请你挑战
1. 这组过滤规则的信噪比是否最优？有没有重要信号会被误杀？有没有噪音会泄漏？
2. "comments >= 3"是一个好的质量代理指标吗？高质量的独立 bug report（0评论但描述精确）会被丢掉。
3. 有没有更好的预过滤策略？比如基于 label 体系、reaction 数量、作者信誉度、Issue 是否被关联到 PR/commit？
4. 不同类型的 Issue 应该用不同的过滤阈值吗？（bug report vs discussion vs incident report）
5. 如果你来设计这个过滤器，你的方案是什么？给出具体的规则和阈值。

---

## 决策点 2：清洗阶段的语义去重

### 当前方案
新判断入库前，与已有判断做语义相似度比较：
- 基于 `when + action` 字段的文本语义相似度
- 阈值 > 0.85 标记为疑似重复
- 由 LLM 最终判断是否真的重复并决定合并策略

### 请你挑战
1. 0.85 这个阈值在什么情况下太高（漏掉重复）？什么情况下太低（误杀相似但不同的判断）？
2. 只比较 `when + action` 够吗？两条 when 和 action 完全不同的判断，可能在语义上是同一条规则的不同表述。
3. 向量相似度是去重的最佳工具吗？还是有更好的方法（如规则化比较：同一 resource + 同一 modality = 疑似重复）？
4. "合并"策略的边界是什么？合并后如何确保不丢失关键信息？
5. 如果你来设计去重系统，你的方案是什么？

---

## 决策点 3：检索阶段的意图展开

### 当前方案
编译引擎收到用户意图后（如"我想用 Python + yfinance 做A股回测系统"），除了直接匹配判断库，还用 LLM 做"意图展开"：
- LLM 思考用户没有提到但必须考虑的问题域
- 每个问题域转化为具体的 query_tags 用于召回额外判断
- 展开上限 5 个问题域
- 只能展开到判断库已覆盖的领域

### 请你挑战
1. "展开上限 5 个"合理吗？对于一个复杂领域（如量化交易），5 个可能不够；对于一个简单任务（如写个脚本抓数据），5 个可能太多。应该动态调整吗？
2. 意图展开是否应该考虑用户的专业水平？新手需要更多展开（他不知道自己不知道什么），专家需要更少（他已经考虑过了）。如何判断用户水平？
3. "只能展开到库中已有判断"这个约束是否太保守？如果库中缺少关键领域的判断，是否应该主动告诉用户"你需要的X领域，当前知识库尚未覆盖"？
4. 展开后的判断如何与直接匹配的判断区分优先级？用户明确提到的（直接匹配）应该比 LLM 推测的（展开匹配）权重更高吗？
5. 有没有替代"LLM 展开"的方法？比如用判断间的关系图谱自动扩展（沿 generates/depends_on 走两跳），而不需要 LLM 猜测？
6. 如果你来设计检索策略，你的方案是什么？

---

## 要求

- 对每个决策点，先说你是否同意当前方案（同意/部分同意/反对）
- 如果反对或部分同意，给出你的替代方案，越具体越好
- 如果你发现了我们完全没考虑到的风险，请直接指出
- 以 markdown 格式输出
```
