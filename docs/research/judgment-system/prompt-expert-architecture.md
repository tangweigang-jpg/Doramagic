## 提示词

```
## 背景

我们正在设计一个"AI顾问"产品——它是一个AI形态的行业专家，用户描述需求后，AI顾问为用户编译出一个"种子晶体"（结构化的知识包），用户把种子晶体带到自己的任何AI工具中使用，AI工具的产出质量会显著提升。

AI顾问的知识来自对大量开源项目的深度代码分析和跨项目合成。种子晶体是可传播的消费品——用户A觉得好用会转发给朋友B。

## 上一轮讨论的结论

上一轮中，我们请四个AI模型（Claude/GPT/Gemini/Grok）各自设计了AI顾问的知识结构框架，然后做了跨模型合成。合成产出了一个包含6种知识类型（世界模型、约束、资源、模式、协议、证据）和4个层级（通用、领域、工程、交叉）的框架。

## 新的反思

在审视合成框架后，我们提出了一个更本质的问题：构成一个行业顶尖专家的核心架构是什么？

我们的假设是三层结构：

### 知识（Knowledge）
专家理解领域的规律。交易日历怎么运作、拆股怎么影响价格序列、回测为什么有幸存者偏差。这是世界模型、约束的理论部分、模式的理论部分。

### 资源（Resources）
专家知道用什么工具，哪些数据源可靠，每个资源的真实能力边界在哪。同样的知识，一个有Bloomberg Terminal的人和一个只有yfinance的人，能做的事完全不同。资源不是知识，是"装备"。而且资源选择会生成约束（选了A就不能做B）。

### 经验（Experience）
专家踩过坑、见过失败、验证过什么有效什么无效。"教科书上说做法A，但在实盘里会遇到问题B"。经验包括：社区的集体经验（开源项目Issue里的踩坑记录、用户论坛里的实盘反馈、种子晶体用户的使用反馈）。经验还提供证据——每条知识来自哪里、置信度多高、被多少人验证过。

### 这三层的关系

之前合成的6种知识类型可以映射到这三层：
- 知识 → 世界模型 + 约束(理论部分) + 模式(理论部分)
- 资源 → 资源
- 经验 → 约束(实践验证) + 模式(实战验证) + 协议 + 证据

一个关键发现：约束和模式各自被拆成了"理论"和"实践"两半。"float不能用于金融计算"可以从数学原理推导（知识），也可以从上线三天后客户投诉中发现（经验）。同一条约束，从经验角度讲对AI的说服力更强。

另一个发现：社区知识自然归入"经验"层——它是集体经验。

## 你的任务

请你从第一性原理审视这个三层模型：

1. **完整性检验**：知识、资源、经验——这三层是否完整覆盖了一个行业顶尖专家所需要的一切？是否有遗漏的第四层？

2. **正交性检验**：这三层是否真正正交（互相独立、不重叠）？还是存在模糊地带？如果有，该怎么处理？

3. **对比检验**：相比6类型框架（世界模型、约束、资源、模式、协议、证据），三层模型是否丢失了重要信息？还是三层是更本质的抽象，6类是其展开形式？

4. **可操作性检验**：如果用这个三层模型来组织AI顾问的知识库、编译种子晶体，它是否足够具体？还是太抽象以至于无法直接指导实现？

5. **你的替代方案**：如果你认为三层不对，你会怎么定义一个行业顶尖专家的核心架构？

用"股票投资分析工具"作为具体场景来验证你的分析。
```
