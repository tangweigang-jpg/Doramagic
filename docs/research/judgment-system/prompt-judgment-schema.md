# 提示词：知识最小单位（Judgment）的元数据技术设计

> 将此提示词分别发给 GPT、Gemini、Grok，收集它们的设计方案

---

## 提示词

```
## 背景

我们正在构建一个AI顾问产品（Doramagic），它为用户编译"种子晶体"——一种结构化的知识包，用户把晶体带到自己的AI工具中使用，AI产出质量显著提升。

经过两轮四方跨模型合成（Claude/GPT/Gemini/Grok），我们确定了以下架构：

### 专家本体（AI顾问的知识库）
由三层构成：
- **知识（Knowledge）**：领域的客观规律、因果关系、不变量
- **资源（Resources）**：工具/API/数据源的真实能力边界，资源选择会生成约束
- **经验（Experience）**：实践验证、失败模式、社区集体踩坑，是知识的实战校准

### 编译引擎（AI顾问的推理过程）
三层碰撞 → 交叉知识 → 决策裁剪 → 输出种子晶体

### 种子晶体（用户消费的蓝图）
包含：世界模型、约束集、资源画像、架构范式、验证协议、证据

## 当前问题：知识的最小单位

我们确定了知识的最小单位叫"判断（Judgment）"，其核心内容结构是：

```
当 [条件] 时，必须/禁止 [判断]，否则 [后果]
```

三个例子：

来自知识层：
> 当进行金融计算时，禁止使用float，否则累计PnL偏差超0.01%

来自资源层：
> 当需要实时交易决策时，禁止依赖yfinance，否则15分钟延迟导致决策失效

来自经验层：
> 当上线前，必须先dry-run 72小时，否则首周出现致命bug（freqtrade社区300+案例）

## 你的任务

请你为这个最小单位设计完整的元数据技术方案。具体来说：

### 1. Schema 设计
为"一颗判断"设计完整的数据结构（字段、类型、枚举值）。需要考虑：
- 核心内容字段（条件、判断、后果）
- 归属标签（属于哪一层、哪个领域、什么适用范围）
- 可信度（来源、置信度、共识度、验证项目）
- 编译用字段（严重度、在晶体中的位置、时效性）
- 关联关系（这颗判断和其他判断的关系——依赖、冲突、生成）

### 2. 关联关系设计
判断之间不是孤立的。比如：
- 选择yfinance（资源判断）→ 生成"不能做实时交易"（约束判断）
- "禁用float"（知识判断）+ "上线三天PnL偏差0.8%"（经验判断）→ 同一约束的两种证据
设计这些关联关系的表达方式。

### 3. 适用范围设计
同一颗判断可能跨领域适用（如float禁令适用于金融、医疗剂量、工程测量），也可能只在特定资源组合下成立。设计"适用范围"的表达机制。

### 4. 版本与时效性
API会废弃、库会升级、规则会变。设计判断的版本管理和时效性标注机制。

## 要求

- 给出具体的schema定义（YAML/JSON Schema/TypeScript type均可）
- 用"股票投资分析工具"的真实例子填充至少5颗判断作为示例
- 考虑编译引擎的查询效率（编译器需要快速找到"与用户意图相关的所有判断"）
- 考虑种子晶体的输出需要（编译器需要把判断组装成晶体的各个section）
- 如果你认为"判断"不是最优的最小单位设计，请提出你的替代方案
- 请以markdown格式输出
```
