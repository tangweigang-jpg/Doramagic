## A. 评估新方案

### 1. 两层模型是否比三层模型更准确？

**是的，但原因不是"三层合并为一层"，而是"维度纠错"。**

旧三层模型按**来源**分层（文档→知识、API→资源、Issue→经验），但来源不等于本质。一条从文档中提取的规则（"用 Decimal"）和一条从 Issue 中提取的规则（"dry-run 72 小时"），在消费端（AI 读晶体时）没有任何区别——都是"当 X 时必须 Y"。按来源分层对生产端有意义（不同 adapter 采集），对消费端没意义。

新两层模型按**功能**分层：蓝图告诉 AI "搭什么结构"，约束告诉 AI "搭的时候守什么规矩"。这两者在消费端是有本质区别的——AI 处理蓝图和处理约束的方式完全不同。

**结论：旧模型分错了维度。新模型的维度更贴近知识被消费的方式。**

### 2. "三层本质相同"是否正确？有反例吗？

**大部分正确，但有一个有意义的反例：工具版本边界。**

"yfinance v0.2.31 开始不再返回 adj_close 字段"——这不是"踩坑后的教训"，而是一个**客观事实**。它不需要有人踩坑才存在，它是 API 变更日志的直接记录。

但这不影响结论，因为在消费端它仍然用同一个句式表达："当使用 yfinance >= 0.2.31 时，禁止访问 adj_close 字段，否则 KeyError"。它是约束层的一部分，只是 freshness=volatile、confidence=1.0。

**结论：三层本质确实相同。个别例外不改变整体判断。合并为"约束层"是正确的简化。**

### 3. 是否遗漏了第三种知识类型？

**可能遗漏了"选型决策知识"——即"为什么选 A 不选 B"。**

以股票投资工具为例：
- 蓝图说："数据采集阶段用 DataProvider 抽象"
- 约束说："yfinance 数据延迟 15 分钟，不能做实时交易"
- 但没有人说："为什么用 yfinance 而不是 Alpaca？因为 yfinance 免费且无需 API key，适合原型阶段"

这种"选型理由"不是蓝图（不影响架构结构），也不完全是约束（不是"当 X 时必须 Y"），而是一种**决策上下文**。

但仔细想，这可以归入约束层的一个变体："当处于原型验证阶段时，应当选择 yfinance 而非 Alpaca，否则增加不必要的注册和配置成本"。句式略牵强，但可以装进约束层。

**结论：暂不需要第三层。选型知识可以用约束句式承载，虽然不完美。如果未来发现大量知识无法用蓝图或约束表达，再考虑扩展。**

### 4. 新方案的潜在盲点和风险

**风险 1：蓝图提取极难自动化。**
约束可以用 LLM 从文档/Issue/代码中提取（已验证）。但蓝图需要**理解整个项目的架构意图**——这远超过"读代码片段提取规则"的难度。第一批蓝图可能需要人工或半自动提取。

**风险 2：蓝图的粒度难以标准化。**
freqtrade 的蓝图应该是 20 行还是 200 行？太粗则没用，太细则变成教程。没有客观的"正确粒度"标准。

**风险 3：蓝图之间可能冲突。**
如果同一个任务有多个蓝图（不同解法），用户或编译引擎需要选择。选择的依据是什么？这引入了一个新的决策层。

**风险 4：约束如何"附着"到蓝图？**
一条约束可能适用于蓝图的某个阶段，也可能跨阶段。"附着"关系的建模不简单。

## B. 蓝图的本质

### 5. 股票投资工具的蓝图示例

```python
"""蓝图：量化回测系统 — freqtrade 模式
来源：freqtrade/freqtrade (22K stars)
解法特征：事件驱动 + 策略插件 + 单交易所
适用场景：个人量化交易，中低频策略（>1分钟级别）
"""

# === 阶段 1：数据管道 ===
class DataProvider:
    """职责：获取和缓存行情数据。只负责"拿数据"，不做分析。"""
    async def fetch_ohlcv(self, pair: str, timeframe: str) -> DataFrame: ...
    async def fetch_orderbook(self, pair: str) -> dict: ...
    # 关键设计：本地缓存层（避免 API 限流）
    # 关键设计：数据校验（缺失值、时间连续性）

# === 阶段 2：策略引擎 ===
class Strategy(ABC):
    """职责：根据数据产生交易信号。纯计算，无副作用。"""
    @abstractmethod
    def populate_indicators(self, df: DataFrame) -> DataFrame: ...
    @abstractmethod  
    def populate_entry_trend(self, df: DataFrame) -> DataFrame: ...
    @abstractmethod
    def populate_exit_trend(self, df: DataFrame) -> DataFrame: ...
    # 关键设计：策略是插件（用户自定义），引擎不关心策略内容
    # 关键设计：信号是 DataFrame 列标记，不是函数调用

# === 阶段 3：交易执行 ===
class Exchange:
    """职责：将信号翻译为交易所指令。处理滑点、重试、部分成交。"""
    async def create_order(self, pair: str, side: str, amount: float) -> Order: ...
    async def cancel_order(self, order_id: str) -> bool: ...
    # 关键设计：交易所是可替换的（ccxt 抽象层）
    # 关键设计：订单状态机（pending → open → filled/cancelled）

# === 阶段 4：风控 ===
class RiskManager:
    """职责：在执行后检查持仓风险。可以强制平仓。"""
    def check_position_size(self, portfolio: Portfolio) -> list[Action]: ...
    def check_drawdown(self, portfolio: Portfolio) -> list[Action]: ...
    # 关键设计：风控在执行之后运行（不是之前）
    # 关键设计：风控有强制平仓权限（覆盖策略信号）

# === 编排 ===
class Bot:
    """主循环：四阶段顺序执行，每个 tick 走一遍。"""
    async def run_tick(self):
        data = await self.data_provider.fetch_ohlcv(...)    # 阶段 1
        signals = self.strategy.populate_entry_trend(data)   # 阶段 2
        orders = await self.exchange.create_order(signals)   # 阶段 3
        actions = self.risk_manager.check_drawdown(...)      # 阶段 4
        for action in actions:
            await self.exchange.execute(action)              # 风控指令执行
```

**蓝图元素：**
- 阶段划分（4 阶段）和执行顺序
- 每个阶段的职责边界（DataProvider 只拿数据不分析）
- 模块间的接口契约（Strategy 输入 DataFrame 输出 DataFrame）
- 关键设计决策（策略是插件、风控有强制权限）
- 数据流向（data → signals → orders → risk actions）

### 6. 蓝图和约束的边界

| 知识 | 蓝图还是约束？ | 判断依据 |
|------|--------------|---------|
| "系统分为数据、策略、执行、风控四个阶段" | **蓝图** | 定义结构 |
| "风控在执行之后运行" | **蓝图** | 定义阶段顺序 |
| "回测时必须使用复权价格" | **约束** | 单条规则，附着在阶段 1 |
| "策略类必须是 ABC 抽象类，用户继承实现" | **边界模糊** → 蓝图 | 它定义了策略模块的扩展机制，属于架构决策 |
| "订单超时 30 秒必须自动取消" | **边界模糊** → 约束 | 它是执行阶段的一条规则，不影响整体架构 |
| "数据获取必须有本地缓存层" | **边界模糊** → 蓝图 | 它决定了数据阶段的内部结构（有缓存 vs 无缓存） |
| "用 ccxt 做交易所抽象层" | **边界模糊** → 蓝图 | 它是一个架构选型，影响执行阶段的设计 |
| "Decimal 不用 float 做金额计算" | **约束** | 跨阶段的规则，不影响架构结构 |

**判断原则：如果移除这条知识后架构结构会变 → 蓝图。结构不变只是质量变 → 约束。**

### 7. 好蓝图 vs 差蓝图

| 维度 | 好蓝图 | 差蓝图 |
|------|--------|--------|
| **粒度** | 最小可运行（15-50 行核心结构） | 要么太粗（5 行伪代码）要么太细（500 行教程） |
| **可扩展** | 明确的扩展点（ABC、Plugin、Hook） | 写死的实现，无法定制 |
| **职责清晰** | 每个模块一句话说清职责 | 模块之间职责纠缠 |
| **数据流可见** | 一眼看出数据从哪来到哪去 | 数据流隐藏在复杂调用链中 |
| **已验证** | 来自真实项目（freqtrade 22K stars） | 理论设计，无实践验证 |
| **解法特征明确** | 说清适用场景和取舍（"中低频、单交易所"） | 号称通用但实际什么都做不好 |

### 8. 蓝图从代码中提取的难点

**难点 1：架构意图 vs 实现细节。** freqtrade 有 10 万行代码，蓝图只需要核心的 50 行。哪 50 行？需要 LLM 理解"什么是架构骨架、什么是实现细节"。

**难点 2：隐含的阶段划分。** 不是所有项目都明确标注"这是阶段 1、这是阶段 2"。阶段划分往往隐含在目录结构、类继承关系、函数调用链中。

**难点 3：多种正确的抽象方式。** 同一个项目可以从不同视角抽象出不同的蓝图。需要选择最有价值的视角。

**可行路径：** 第一批蓝图半自动提取——人给出项目和任务描述，LLM 读取核心代码后生成蓝图草稿，人审核修正。随着蓝图积累，训练提取模式。

## C. 挑战假设

### 9. 蓝图是否也是约束的特殊形式？

**不是。** 这是两者最关键的区别：

- 约束是**否定式**知识："不准做 X"
- 蓝图是**肯定式**知识："应该搭成这样"

你可以把所有约束去掉，蓝图仍然是一个可运行的骨架（只是容易踩坑）。但你不能把蓝图去掉只留约束——AI 拿到一堆"不准做 X"的规则，不知道"那应该做什么"。

**蓝图和约束是不可互相还原的。两层不能合并为一层。**

### 10. 是否需要三层或更多层？

如果我来重新设计，我会用两层，但给约束层加**标签维度**而非硬分层：

```
蓝图层：架构结构（不变）

约束层：统一存储，但每条约束带标签：
  - scope: universal / domain / project
  - confidence: 0.0 - 1.0
  - source_type: code_analysis / community / doc / changelog
  - applies_to: blueprint_stage_id（附着到蓝图的哪个阶段）
  - freshness: stable / semi_stable / volatile
```

这比硬分三层更灵活——你可以按任何维度查询和排序，而不是被预定义的层限制住。

### 11. 跨平台需求对架构的约束

种子晶体要在 Claude Code、Cursor、Codex、普通 ChatGPT 中都有效，这意味着：

- **蓝图必须是纯代码**（不是某个平台的 DSL）——Python/TypeScript 代码任何 AI 都能读
- **约束必须是自然语言 + 结构化**——不能依赖某个平台的特殊指令格式
- **context_acquisition 必须平台无关**——不能写"查看用户的 CLAUDE.md"，而是写"查阅用户的项目配置和历史偏好"
- **晶体格式必须是 Markdown**——所有 AI 平台都能解析 Markdown

### 12. 根本性错误检查

**我没有发现根本性错误。** 但有一个值得警惕的隐患：

蓝图的价值依赖于"用户的任务能匹配到某个已有蓝图"。如果用户的需求是全新的、没有对应开源项目的，Doramagic 就无法提供蓝图，晶体退化为一堆散装约束。

应对方式：对无蓝图覆盖的任务，诚实告知用户"我们在这个领域没有架构蓝图，只能提供约束规则"。不伪装能力边界。

## D. 竞争对手视角

### 13. 竞品如何反击

**Cursor/Copilot 的反击：** "我们直接用 RAG 检索用户 codebase 中的架构模式，不需要预编译蓝图。" 这确实是一个威胁——如果用户已经有了一个架构良好的 codebase，他们不需要 Doramagic 的蓝图。

**Doramagic 的回应：** 但大部分用户是从零开始的——他们问"帮我做一个量化系统"的时候，codebase 是空的，RAG 没东西可检索。Doramagic 的价值恰恰在"从零到一"的阶段。

**Devin 的反击：** "我直接执行，不需要给用户一个配方让他自己做。" 这是路线之争——Doramagic 赌的是"用户想要控制权"，Devin 赌的是"用户想要结果"。

### 14. 护城河分析

| 要素 | 可复制性 | 说明 |
|------|---------|------|
| 约束规则 | **易复制** | 任何人都能写 LLM prompt 从文档中提取规则 |
| 蓝图 | **中等难复制** | 需要深度代码分析 + 架构理解 + 人工审核 |
| 蓝图 + 约束的组合 | **难复制** | 约束正确附着到蓝图的正确位置，需要领域理解 |
| 蓝图库的广度 | **很难复制** | 覆盖 50+ 领域 × 每领域 3-5 个蓝图 = 需要大量积累 |
| 跨项目综合蓝图 | **极难复制** | 从 3 个项目中提炼出一个综合蓝图，需要架构判断力 |

**真正的护城河不是单个蓝图，而是蓝图库的广度 × 约束的附着精度。** 这需要时间积累，不是写一个好 prompt 就能复制的。

## 总结

1. **两层模型（蓝图 + 约束）是正确的方向。** 旧三层模型分错了维度。
2. **蓝图是核心价值，约束是增值服务。** 没有蓝图的约束价值有限，没有约束的蓝图也能用。
3. **最大风险是蓝图提取的自动化程度。** 第一批需要半自动，长期需要解决。
4. **约束层不再硬分三层，改用标签维度。** 更灵活，消费端按需查询。
5. **蓝图和约束不可互相还原。** 两层架构是稳定的，不会退化为一层。
