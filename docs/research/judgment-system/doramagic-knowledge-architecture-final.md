这三层不能混在一起。专家本体回答"顾问脑子里有什么"，编译引擎回答"顾问怎么思考"，种子晶体回答"用户拿到什么"。

## 一、专家本体：知识 + 资源 + 经验

这是 AI 顾问的认知架构，也是 Doramagic 知识库的组织方式。

### 知识（Knowledge）—— 世界如何运转

领域的客观规律、因果关系、不变量。可以从理论推导，不需要外部验证就能成立。

```
股票领域示例：
├── 市场时间非连续（周末/节假日无数据，竞价机制）
├── 价格属离散有理数空间，非连续实数
├── 企业行为（拆股/分红）改变价格序列语义
├── 财报发布日期晚于所属季度结束日
├── IEEE 754 浮点数无法精确表示大多数十进制分数
└── 幸存者偏差：历史数据只含当前仍上市的股票
```

### 资源（Resources）—— 手里有什么装备

工具、API、数据源、开源库的真实能力边界。每个资源自带约束——选择一个资源就意味着接受它的限制。

```
股票领域示例：
├── yfinance: 免费历史EOD → 不能做实时交易决策
├── Alpaca: 免费实时+执行 → 仅美股，PDT规则适用
├── TA-Lib: C库指标计算 → 部署需编译工具链
├── pandas float: 默认数值类型 → 必须改Decimal做金融计算
└── Tushare Pro: A股全量 → 付费积分制，需预算管理
```

### 经验（Experience）—— 战场上学到的

实践验证、失败模式、社区集体踩坑、边界案例。经验是知识的实战校准——同一条约束，有经验佐证的比纯理论推导的更可信。

```
股票领域示例：
├── freqtrade Issue #2345: float累计误差上线三天达0.8%
├── zipline社区共识: 必须先dry-run 72小时再live
├── 量化论坛300+帖: pandas rolling做回测易引入look-ahead
├── vnpy用户反馈: A股涨停板场景策略无法执行
└── 多项目共识: 回测必须建模slippage+commission
```

### 三者的关系

```
知识提供理论 ──→ 经验提供验证 ──→ 知识获得置信度
资源提供能力 ──→ 资源选择生成约束 ──→ 约束回流到知识
经验提供教训 ──→ 教训指导资源选择 ──→ 资源获得优先级
```

三者不是静态并列，而是持续互相校准。

## 二、编译引擎：三层碰撞的决策过程

这是 AI 顾问的推理过程——把原材料变成种子晶体。

```
输入: 用户意图 + 用户上下文
         │
         ▼
    ┌─────────────┐
    │  意图理解     │  这是分析工具还是交易系统？
    │  上下文探测   │  个人学习还是生产环境？
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │  知识激活     │  匹配领域的世界模型和理论约束
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │  资源匹配     │  推荐资源组合 → 每个资源生成约束
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │  经验校准     │  用实践证据给每条约束打置信度
    │              │  补充理论未覆盖的实战暗知识
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │  交叉碰撞     │  知识×资源×经验 → 产出交叉约束
    │              │  （如：时间缺口×时序计算）
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │  决策裁剪     │  根据上下文筛选约束子集
    │              │  约束排序（致命→生产→建议）
    │              │  冲突消解、降级策略
    └──────┬──────┘
           ▼
    ┌─────────────┐
    │  结晶输出     │  打包为种子晶体
    └─────────────┘
```

编译引擎不是第四种原材料，是加工过程。

## 三、种子晶体：用户消费的蓝图

种子晶体是三层碰撞后的结晶产物。用户不需要看到原材料，只需要拿到蓝图。

### 晶体结构

```yaml
crystal:
  # ── 元信息 ──
  meta:
    intent: "股票投资分析工具"
    context: "个人学习"
    version: "1.0"
    source_projects: [freqtrade, zipline, vnpy]
    freshness: "semi-stable, review in 6 months"

  # ── 来自知识层 ──
  world_model:
    domain: equities
    physics:
      - "市场时间非连续（周末/节假日/竞价机制）"
      - "价格属离散有理数空间"
      - "财报发布日晚于所属季度"
      - "企业行为改变价格序列语义"

  # ── 来自资源层 ──
  resources:
    recommended:
      - name: yfinance
        for: 历史数据回测
        constraints_generated:
          - "不能做实时交易决策"
          - "必须本地缓存防限频"
        alternatives: [alpaca, tushare_pro]
    stack: [python, pandas, duckdb]

  # ── 来自知识×经验碰撞 ──
  constraints:
    fatal:
      - rule: "财务计算禁用float，必须用Decimal"
        consequence: "累计PnL偏差"
        verify: "静态扫描float在Decimal应出现的上下文"
        evidence: "S2共识 + freqtrade Issue #2345"
      - rule: "禁止回测中的未来函数(look-ahead)"
        consequence: "策略虚假盈利"
        verify: "所有indicator必须.shift(1)"
        evidence: "S2共识: freqtrade+zipline+vnpy"
      - rule: "禁止忽略幸存者偏差"
        consequence: "回测收益虚高3-8%"
        verify: "验证样本是否含退市股"
        evidence: "S2共识 + 量化社区"
    production:
      - rule: "交易状态必须有不可变审计日志"
      - rule: "回测必须建模slippage+commission"
    advisory:
      - rule: "回测绑定数据快照版本"
      - rule: "优先使用事件驱动回测引擎"

  # ── 来自知识×资源×经验的交叉碰撞 ──
  architecture:
    blueprint: "ingest → normalize → feature → simulate → evaluate → report"
    key_decisions:
      - "技术指标基于交易日序列计算，不插值非交易日"
      - "raw价格和adjusted价格分开存储"
      - "基本面因子必须point-in-time join"
    patterns:
      - name: "Research/Execution分离"
        description: "研究用向量化快速迭代，执行用事件驱动"

  # ── 来自经验层 ──
  protocols:
    before_deploy:
      - "必须先dry-run模式验证72小时"
      - "必须跑out-of-sample + walk-forward测试"
    runtime:
      - "数据新鲜度监控"
      - "API限流告警"
    reporting:
      - "报告必须包含：数据截止日、基准、费用假设"

  # ── 来自资源缺失时的降级路径 ──
  degradation:
    if_no_historical_constituents:
      action: "降级为当前股票池历史观察"
      disclosure: "报告声明样本构造方式"
    if_no_realtime_api:
      action: "仅支持EOD分析"
      disclosure: "禁用盘中功能"

  # ── 来自经验层的溯源信息 ──
  evidence:
    consensus: [float禁令, 未来函数检测, 费用建模, dry-run协议]
    disputed: [事件驱动 vs 向量化回测]
    source_confidence:
      S2_cross_project: high
      S3_community: medium-high
      S4_reasoning: medium
```

## 四、普适性

这个架构对任何领域都成立，只需替换内容：

| 组成 | 股票投资工具 | 智能家居系统 | 医疗影像分析 |
|------|-------------|-------------|-------------|
| 知识 | 交易日历、复权、监管规则 | 设备响应延迟、无线信号覆盖 | 影像分辨率、解剖结构、病理特征 |
| 资源 | yfinance/Alpaca/TA-Lib | HomeAssistant/ZigBee/MQTT | DICOM/PyTorch/RadImageNet |
| 经验 | 社区踩坑、实盘验证 | 用户论坛、兼容性问题 | 临床验证、误诊案例 |
| 编译 | 意图理解→碰撞→裁剪→输出 | 同 | 同 |
| 晶体 | 约束+架构+协议+降级 | 同结构，不同内容 | 同结构，不同内容 |

框架是骨架，领域知识是肉。骨架不变，肉随领域换。

## 五、Doramagic 的三件事

| # | 做什么 | 对应架构层 | 一次性/持续 |
|---|--------|-----------|-----------|
| 1 | 深度分析开源项目，提取知识+资源+经验 | 专家本体 | 持续积累 |
| 2 | 构建编译引擎，实现三层碰撞+决策裁剪 | 编译引擎 | 一次性构建，持续优化 |
| 3 | 为每个领域编译种子晶体，供用户传播使用 | 种子晶体 | 按领域编译，按反馈迭代 |

核心飞轮：晶体被使用 → 用户反馈 → 经验层增长 → 晶体迭代 → 更多人使用
