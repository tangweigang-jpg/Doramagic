# AI 顾问知识结构框架 v0.2

> 场景：用户说"帮我做一个股票投资分析工具"
> 目的：定义 AI 顾问需要的完整知识结构，用于编译种子晶体
> 变更记录：v0.1 → v0.2 增加知识来源方法论、可移植性、时效性、用户上下文适配；
>   将"差距=约束"标注为假设而非结论

---

## 0. 核心假设（待验证）

本框架基于以下假设构建。如果假设被推翻，框架需要根本性修改。

**假设 H1：LLM 和生产级产出之间的差距，主要是约束知识的缺失。**
依据：A/B 实验中，注入约束后生产特性命中率从 0/8 提升到 7/8。
待检验：是否存在约束之外的知识维度同样重要？（比如：领域心智模型、决策优先级、资源组合判断）

**假设 H2：种子晶体作为知识载体，能在不同用户和不同 AI 工具间保持有效性。**
依据：种子晶体是"对象层知识"（描述世界是什么样），而非"控制层指令"（命令 AI 做什么），理论上不依赖特定 AI 的行为模式。
待检验：不同 LLM 对同一晶体的遵守率是否有显著差异？

**假设 H3：深度代码分析 + 跨项目合成能产出 LLM 自身无法推理出的知识。**
依据：深度分析 vs README 对比实验，知识提取量差 750%。
待检验：这些额外知识中，有多少比例会真正改变 LLM 的产出？

## 1. 知识模型：三个维度

AI 顾问的知识结构有三个正交维度：类型、层级、来源方法。

### 维度一：知识类型（顾问知道什么）

```
┌─────────────────────────────────────────────┐
│  约束（Constraint）                          │
│  什么不能做，以及为什么                        │
│  → 直接改变 LLM 的产出行为                    │
├─────────────────────────────────────────────┤
│  资源（Resource）                             │
│  用什么来做，每个资源的真实能力边界             │
│  API、开源库、数据源、第三方服务               │
│  → 资源选择会生成约束（关联关系）              │
├─────────────────────────────────────────────┤
│  模式（Pattern）                              │
│  验证过的做法，以及它适用的边界条件             │
│  → 模式是约束的正面表达（"应该这样做"）        │
├─────────────────────────────────────────────┤
│  心智模型（Mental Model）[待验证]             │
│  领域专家如何思考问题的框架                    │
│  → 可能是 H1 之外的重要维度                   │
│  → 例：量化交易者先想风险再想收益               │
└─────────────────────────────────────────────┘
```

**关于"心智模型"：** 这是一个不确定的类型。约束告诉 LLM "不要做X"，但可能还需要告诉 LLM "像领域专家一样思考"——比如，量化交易者在设计系统时首先考虑的不是"怎么赚钱"而是"怎么控制风险"。这种思维框架是否独立于约束？还是可以被约束覆盖？需要 GPT/Gemini 的视角来验证。

### 维度二：知识层级（知识的来源深度）

```
┌─────────────────────────────────────────────┐
│  L0: 通用知识（LLM 自带，不提供）             │
│  Python, REST API, DataFrame, 基础金融概念    │
│  ⚠ 边界模糊：LLM 可能"知道"Decimal 但生成时   │
│    仍用 float——这算 L0 还是需要外部注入？      │
├─────────────────────────────────────────────┤
│  L1: 行业知识（领域特异）                     │
│  金融数据源、监管规则、指标陷阱、市场微观结构   │
├─────────────────────────────────────────────┤
│  L2: 系统知识（工程架构）                     │
│  可靠性、数据管道、API 集成、错误处理          │
├─────────────────────────────────────────────┤
│  L3: 交叉知识（L1×L2 碰撞产物）              │
│  行业特性如何改变系统设计决策                   │
│  ★ 这是种子晶体的核心价值层 ★                 │
│  ★ 也是跨项目合成的核心产出 ★                 │
└─────────────────────────────────────────────┘
```

**关于 L0 边界：** LLM "知道"某件事和"在生成代码时实践"这件事是两回事。LLM 被问"金融计算应该用什么类型？"会回答 Decimal，但在生成股票工具时仍然用 float。这意味着某些知识虽然在 L0，但需要作为约束重新注入才能激活。种子晶体的作用不仅是提供新知识，还有**激活 LLM 已有但沉睡的知识**。

### 维度三：知识来源方法（知识怎么来的）

这不是附录，而是框架的核心维度。来源方法决定了知识的可信度和独特性。

```
┌─────────────────────────────────────────────┐
│  S1: 单项目深度分析                           │
│  全量代码阅读 → 提取架构决策、约束、资源选择   │
│  产出：项目特异的 L1/L2 知识                  │
├─────────────────────────────────────────────┤
│  S2: 跨项目合成                               │
│  多个项目在同一问题上的做法对比                 │
│  ALIGNED → 行业共识（高置信度约束）            │
│  DIVERGENT → 争议点（需人工判断或更多数据）    │
│  产出：L3 交叉知识（框架的核心价值）           │
├─────────────────────────────────────────────┤
│  S3: 社区信号挖掘                             │
│  Issue/PR/Discussion 中的踩坑记录              │
│  高 Issue/Star 比 = 痛点密集区                │
│  Reverted PR = 已验证的失败路径               │
│  产出：failure 类知识（约束的经验证据）         │
├─────────────────────────────────────────────┤
│  S4: AI 顾问推理                              │
│  在充分理解 L1+L2 后，推理 L3 交叉约束        │
│  ⚠ 置信度最低，必须标注为推理而非实证          │
└─────────────────────────────────────────────┘
```

### 三维交叉矩阵

种子晶体的每条知识都可以定位在这个三维空间中：

|               | L1 行业 | L2 系统 | L3 交叉 |
|---------------|--------|--------|--------|
| **约束**       | 监管规则、指标陷阱 | 幂等性、精度要求 | 时间缺口×时序计算 |
| **资源**       | 数据API、行情源 | 基础库、框架 | 资源组合的隐含约束 |
| **模式**       | 策略回测流程 | circuit breaker | 时间感知限频 |
| **心智模型?**  | 风险优先思维 | 防御性编程 | ? |

每个单元格的知识还有来源标签（S1/S2/S3/S4），标注置信度。

## 2. L1: 行业知识 — 金融/股票领域

### L1-A: 数据源（资源+约束）

AI 顾问必须知道每种数据源的真实能力边界，而非文档宣称的能力。

**资源地图（用什么来做）：**

| 资源 | 能力 | 真实边界 | 选择后产生的约束 | 来源 |
|------|------|---------|----------------|------|
| yfinance | 免费历史OHLCV | 无SLA，Yahoo随时改格式，15min延迟 | 只能用于回测，不能实时交易 | S2: freqtrade/zipline 均不用于实时 |
| Alpaca API | 免费实时+交易执行 | 仅美股，PDT规则适用 | 必须处理PDT限制，必须管理API key | S1: Alpaca文档+社区反馈 |
| Tushare Pro | A股全量数据 | 付费积分制，免费额度有限 | 必须实现积分预算管理 | S1: vnpy 集成经验 |
| IEX Cloud | 实时+历史 | 按消息计费，成本不可预测 | 必须实现调用计量和预算告警 | S3: Issue 中的费用投诉 |
| TA-Lib | 技术指标计算 | C库依赖，安装复杂 | 部署环境必须有编译工具链，或改用纯Python替代(pandas-ta) | S2: 多项目分歧点 |

**约束（什么不能做）：**

| 知识点 | 来源 |
|--------|------|
| 免费 API 不能作为交易决策的唯一数据源 | S2: 多项目踩坑共识 |
| 历史数据的幸存者偏差问题 | S2: 量化社区共识 |
| 不同数据源的OHLCV可能不一致（调整价vs未调整价） | S2: 跨项目对比发现 |

### L1-B: 监管/规则约束

不同市场的交易规则差异，违反后果严重。

| 知识点 | 来源 | 时效性 |
|--------|------|--------|
| PDT 规则（美股日内交易限制） | 监管文档 | 稳定（法规级，变更频率低） |
| T+1/T+2 结算周期对资金计算的影响 | 行业惯例 | ⚠ 美股 2024 年从 T+2 改为 T+1 |
| A 股涨跌停板对策略执行的影响 | 市场特异规则 | 稳定 |

### L1-C: 指标/策略陷阱

LLM 最容易出错的地方——知道指标公式但不知道使用陷阱。

| 知识点 | 来源 |
|--------|------|
| RSI 在趋势市中长期超买/超卖不回归 | S3: 回测社区反复讨论 |
| MACD 在震荡市产生大量虚假信号 | S3: 回测社区反复讨论 |
| 回测过拟合：样本内完美，样本外崩溃 | S2: 量化框架文档共识 |

## 3. L2: 系统知识 — 工程架构

### L2-A: 数据管道（资源+约束+模式）

**资源：**

| 资源 | 适用场景 | 选择后产生的约束 | 来源 |
|------|---------|----------------|------|
| SQLite | 单用户本地分析 | 不能并发写入，数据量上限~100GB | S2: 多项目默认选择 |
| PostgreSQL+TimescaleDB | 时序数据存储 | 需要运维数据库服务 | S1: freqtrade 推荐 |
| Redis | 实时行情缓存 | 数据非持久化（需配合持久层） | S2: 实时系统共识 |
| Celery/APScheduler | 定时任务调度 | Celery需要broker，APScheduler适合单进程 | S2: 分歧点 |

**约束：**

| 知识点 | 来源 |
|--------|------|
| 外部 API 调用必须有 circuit breaker | S2: 工程最佳实践共识 |
| 指数退避 + jitter 防止 API 限频风暴 | S2: 多项目共识 |
| 数据管道必须幂等（重跑不产生重复） | S2: 工程最佳实践共识 |

### L2-B: 计算精度（约束）

| 知识点 | 来源 | L0 边界问题 |
|--------|------|------------|
| 财务计算用 Decimal，不用 float | S2: 金融工程共识 | ⚠ LLM 知道但不执行——需要约束激活 |
| 百分比计算用基点(bps)避免精度陷阱 | 行业惯例 | LLM 通常不知道 |

### L2-C: 状态管理与持久化（约束+模式）

| 知识点 | 来源 |
|--------|------|
| 交易状态必须有审计日志（immutable） | 合规要求 |
| 策略参数和交易记录必须分离存储 | S2: 架构最佳实践 |
| 异常中断后必须能从上次状态恢复 | S2: 可靠性共识 |

## 4. L3: 交叉知识 — 种子晶体的核心

### 这一层的知识从何而来？

交叉知识不存在于任何单一来源。它产生于：
- S2 跨项目合成：多个项目在同一问题上的不同处理方式
- S4 AI 推理：在充分理解 L1+L2 后，推导行业特性对系统设计的影响
- S3 生产事故：Issue 中的 bug 报告，往往暴露 L1×L2 的碰撞点

### 股票投资工具的交叉知识示例

#### X1: 时间缺口 × 时序计算
- **L1 行业事实**: 股市周末/节假日不开盘，数据有天然缺口
- **L2 系统默认**: 时序库（pandas resample）默认连续时间轴，会插值填充
- **L3 交叉约束**: 技术指标（MA/RSI/MACD）必须基于交易日序列计算，不能插值非交易日。但"持仓天数"计算需要用日历日（影响税务）。同一个工具里必须维护两套时间轴。
- **违反后果**: 5日均线变成7日均线（含周末），所有技术信号失真
- **来源**: S2 (freqtrade 和 zipline 都维护 trading_calendar) + S4 推理

#### X2: API 限频 × 市场时间窗口
- **L1 行业事实**: 开盘前30分钟和收盘前15分钟是数据请求高峰
- **L2 系统默认**: 固定间隔拉取 + 统一 rate limiter
- **L3 交叉约束**: rate limiter 必须是时间感知的——高峰期自动降频或切换备用数据源，非高峰期更激进拉取
- **违反后果**: 开盘时被限频，错过关键价格变动
- **来源**: S3 (多个项目 Issue 中报告开盘时数据缺失) + S4 推理

#### X3: 回测引擎 × 幸存者偏差
- **L1 行业事实**: 历史股票数据通常只包含当前仍上市的股票
- **L2 系统默认**: 从 API 拉取当前股票列表，用历史数据回测
- **L3 交叉约束**: 回测引擎必须使用"时点成分股"数据
- **违反后果**: 策略回测年化收益虚高3-8个百分点
- **来源**: S2 (zipline 有 survivorship-bias-free 数据集讨论) + S1

#### X4: 实时数据 × 状态恢复
- **L1 行业事实**: 股票数据是连续流，中断后需要补全
- **L2 系统默认**: 进程崩溃后从头拉取
- **L3 交叉约束**: 区分"冷启动"和"热恢复"——热恢复补全缺失段后必须重算有状态指标（如EMA依赖前值）
- **违反后果**: 有状态指标在恢复后跳变，产生虚假信号
- **来源**: S4 推理（置信度中等，需实证验证）

#### X5: 多市场 × 时区 × 汇率
- **L1 行业事实**: 用户可能同时跟踪美股和A股
- **L2 系统默认**: 所有时间戳统一为 UTC
- **L3 交叉约束**: UTC 存储正确，但展示层按市场切换时区。跨市场比较（如 AH 股溢价）时，汇率时间戳必须与股价时间戳对齐到同一粒度
- **违反后果**: AH 溢价计算在汇率波动日偏差 2-3%
- **来源**: S4 推理 + S1 (vnpy 多市场模块分析)

## 5. 可移植性：种子晶体如何跨用户、跨工具生效

种子晶体是可传播的消费品。用户 A 用了觉得好，转发给朋友 B。B 用不同的 AI 工具（可能是 Claude、GPT、Gemini、本地模型）。同一个晶体必须都有效。

### 可移植性设计约束

**P1: 不依赖特定 LLM 的指令格式。**
晶体中不写"你是一个XXX助手"或"请按以下步骤执行"（控制层指令）。只写事实性约束（对象层知识）。事实不依赖接收者是谁。

**P2: 不假设用户的技术水平。**
同一个晶体可能被开发者用来写代码，也可能被非技术用户交给 AI 让 AI 全权处理。约束的表述应该面向 AI 消费，而不是面向人类阅读。

**P3: 资源推荐需要提供替代方案。**
用户 A 在中国，用 Tushare；用户 B 在美国，用 Alpaca。晶体不能绑定单一资源，必须提供"如果用 X 则约束是 Y，如果用 Z 则约束是 W"的条件结构。

**P4: 约束按严重度分级。**
不同复杂度的需求不需要遵守所有约束。个人学习项目可以忽略审计日志；生产交易系统不能。晶体需要标注每条约束的适用场景。

### 用户上下文适配

同一个"股票投资分析工具"请求，不同用户上下文需要不同的约束子集：

| 用户上下文 | 优先约束 | 可忽略约束 |
|-----------|---------|-----------|
| 个人学习/模拟 | 数据精度、指标陷阱 | 审计日志、高可用、合规 |
| 个人实盘交易 | 全部 L1 监管约束、资金安全 | 高并发、多市场 |
| 团队/生产系统 | 全部约束 | 无 |

**编译时决策：** AI 顾问在编译种子晶体时，需要根据用户上下文筛选约束集。这意味着编译流程需要一个"上下文感知"步骤。

## 6. 时效性：知识会过期

### 知识的保鲜分类

| 时效类别 | 典型寿命 | 示例 | 处理方式 |
|---------|---------|------|---------|
| 稳定 | 3年+ | 设计模式、算法原理 | 写入晶体，无需更新机制 |
| 准稳定 | 6-18个月 | 框架最佳实践、API用法 | 写入晶体，标注版本，定期审查 |
| 易变 | 1-6个月 | API 限频策略、费用结构 | 写入晶体但标注 ⚠ 时效敏感 |
| 高度易变 | <1个月 | 具体股价、市场状态 | **不写入晶体**，由用户的 AI 实时获取 |

### 示例

- "yfinance 无 SLA" → **稳定**（这是结构性问题，不会改变）
- "美股结算周期 T+1" → **准稳定**（2024 年刚从 T+2 改为 T+1，短期不会再变）
- "IEX Cloud 按消息计费" → **易变**（定价策略随时可能调整）
- "特斯拉当前股价" → **高度易变**（绝不写入晶体）

## 7. 编译流程：从知识到种子晶体

```
用户意图: "股票投资分析工具"
    │
    ▼
用户上下文探测: 个人学习？实盘交易？团队生产？
    │
    ▼
意图分解: [数据获取, 技术分析, 回测, 可视化, 风险管理]
    │
    ▼
资源推荐: 每个子意图 → 推荐资源组合 + 替代方案 + 每个资源带来的约束
    │
    ▼
约束收集: 资源产生的约束 + 行业约束 + 系统约束 + 交叉约束
    │
    ▼
上下文筛选: 根据用户上下文过滤不适用的约束
    │
    ▼
优先级排序: 按 consequence 严重度排序
    │
    ▼
时效性标注: 每条知识标注保鲜类别
    │
    ▼
结构化输出: 种子晶体
  ├── 适用场景声明（这个晶体为谁编译、适用什么上下文）
  ├── 推荐资源栈（附选择理由、替代方案、每个资源的约束链）
  ├── 硬约束（按严重度排序，附后果说明 + 来源标注 + 时效标注）
  ├── 验收标准（可检查的条件）
  └── 可选：最小代码骨架（仅展示约束如何体现）
```

## 8. 自我质疑：这个框架可能错在哪？

### Q1: "差距=约束"是否完整？
可能遗漏：决策优先级知识。不是"不能做X"，而是"在 A 和 B 之间，应该优先考虑 A"。比如量化系统中"风险控制优先于收益优化"——这不是约束，是决策框架。

### Q2: 四层分级是否是最优分法？
可能的替代：按"用户旅程阶段"分——数据获取阶段的知识、分析阶段的知识、执行阶段的知识。这种分法可能更贴近种子晶体的消费场景。

### Q3: 跨项目合成是否总是产出更好的知识？
可能的反例：一个领域只有一个高质量项目时，跨项目合成反而引入噪音。合成的前提是多个项目都达到一定质量门槛。

### Q4: 种子晶体的大小是否有注意力上限？
之前的研究表明 SKILL.md 在 80 行以上注意力衰减。种子晶体是否有类似限制？如果有，大型需求的晶体如何拆分？
