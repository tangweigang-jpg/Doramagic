# Gemini 的 AI 顾问知识结构框架

> 来源：Gemini (Antigravity)，2026-04-02

---

## 核心认知修正

Gemini 认为"差距=约束缺失"非常敏锐但不完整。

差距不仅是"不知道什么不能做"（约束/Defense），还缺失：
- "必须按照什么骨架去做"（领域本体与共识建设/Offense/Structure）
- "选择A意味着放弃B"（基于资源映射的生成式决策/Trade-offs）

种子晶体应作为"领域同构层"而存在。

## 知识类型（4类）

1. **领域本体 (Ontology)**: 业务对象在真实物理世界里的规律
2. **绝对约束 (Fatal Constraints)**: 突破即导致系统失效或致命副作用的边界（红线）
3. **共识范式 (Consensus Patterns)**: 行业内为应对某种物理规律而形成的"标准解法"
4. **生成式权衡 (Trade-off Generators)**: 基于资源限制导致的"连锁决策"

## 知识层级（3层）

1. **领域物理层 (Domain Physics)**: 完全独立于软件之外的金融事实
2. **系统架构层 (System Architecture)**: 软件如何承载这些事实
3. **组件实现层 (Component Implementation)**: 具体的函数、库、API怎么写

## 交叉矩阵

|  | 领域本体 | 绝对约束 | 共识范式 | 生成式权衡 |
|---|---|---|---|---|
| **领域物理层** | 时间非连续性（周末/节假日/竞价机制） | 幸存者偏差、未来函数/偷价 | 前复权/后复权处理 | 交易频率权衡 |
| **系统架构层** | 信号与执行解耦 | 回测引擎禁止同步阻塞查询 | 事件驱动回测 | 资源约束传导(如API→Rate Limiter) |
| **组件实现层** | 价格属离散有理数空间 | float禁令、.shift(1)强制 | UTC统一+本地时区展示 | 存储介质权衡 |

## 知识间的生成关系

Gemini 强调知识必须是"决策图谱"(DAG/AST)，不能是静态列表。

典型生成链：
1. 用户诉求 → 引入资源需求（如"盘中盯盘"→需要WebSocket行情）
2. 资源物理限制 → 架构选择（必须异步事件驱动）
3. 架构选择 → 绝对约束（异步回调内严禁阻塞I/O）

## LLM已有 vs 必须注入

| 能力 | LLM自有 | 晶体注入 |
|---|---|---|
| 算法与库 | 知道EMA/MACD公式、pandas API | 特定库引发的隐性漏洞（如滚动预测必须dropna） |
| 架构组件 | 知道Redis/REST/WebSocket | 领域组件组合共识（如发单模块必须用低延迟队列） |
| 业务逻辑 | 知道Buy/Sell | 业务物理学陷阱（如A股T+1、涨跌停无法成交） |
