# Blueprint 架构四方综合裁决

> 参与模型：Claude Opus 4.6 / GPT-4o / Gemini 2.5 Pro / Grok
> 日期：2026-04-04
> 裁决人：Claude（综合四方意见，标注共识与分歧）

---

## 一、全票共识（4/4 一致）

### 1. 两层模型优于旧三层模型
四个模型一致认为"蓝图 + 约束"比"知识 + 资源 + 经验"更准确。
- **Claude**：旧模型按来源分层，新模型按消费方式分层，后者更正确
- **GPT**：旧三层回答"知识从哪来"，新两层回答"AI 该怎么用"
- **Gemini**：新模型契合编译器理论——蓝图=AST，约束=语义检查器
- **Grok**：旧模型导致交叉污染，新模型正交

### 2. 蓝图和约束不可合并
四个模型一致认为两层不能退化为一层。
- **核心论据**：蓝图是结构型知识（肯定式："搭成这样"），约束是谓词型知识（否定式："不准这样"）。把蓝图拆成一堆约束会丧失整体形态表达能力。
- **Gemini 补充**：LLM 对模板/示例的吸收力远大于对冗长规则集的吸收力。蓝图作为结构主轴最顺应注意力机制。

### 3. 旧三层"本质相同"基本正确
四个模型都同意大部分规则本质是"踩坑后的教训"，但均指出存在少量反例：
- **纯客观事实**（API 文档写死的能力边界、IEEE 754 精度特性、市场交易规则）
- 这些反例不改变结论：合并为约束层是正确简化，用标签（confidence/scope）区分即可

### 4. 蓝图提取极难自动化
四个模型一致警告蓝图提取的难度远高于约束提取。
- 第一批蓝图需要人工或半自动
- 纯 LLM 读代码提取蓝图准确率低（Grok 估计 <40%）
- Gemini 建议必须用 AST + Call Graph 作为 LLM 前置处理器

## 二、多数共识（3/4 一致）

### 5. 需要"验收标准"作为独立组成部分
- **Grok**：验收协议（Verification Protocol）是致命遗漏，必须立即补上
- **GPT**：Acceptance Checks 应作为正式组成部分
- **Gemini**：验证矩阵（Verification Matrix）是四象限之一
- **Claude**：未将其作为独立层提出（当前晶体格式已包含验收标准段落）

**裁决**：验收标准在当前晶体格式中已存在（第三段），但在知识存储层面没有独立建模。需要确认：验收标准是从蓝图+约束自动派生，还是需要独立采集和存储。

### 6. 蓝图不能过度"代码模板化"
- **GPT**（强烈警告）：蓝图过于靠近代码模板会绑定语言/框架，降低跨平台传播性
- **Gemini**（强烈警告）：好蓝图只提供脚手架和类型签名，差蓝图是项目源码的 Copy-Paste
- **Grok**：蓝图应是最小可运行骨架，但承认过度具体的风险
- **Claude**：提出 15-50 行核心结构，未充分警告过度具体的风险

**裁决**：蓝图应偏向"接口 + 数据流 + 阶段划分"，而非"可运行代码模板"。GPT 的七元素框架（目标边界、核心阶段、关键实体、模块接口、数据流、状态点、可替换点）比纯代码骨架更稳健。

## 三、显著分歧

### 7. 是否需要"上下文契约"作为独立组成部分

| 模型 | 立场 | 理由 |
|------|------|------|
| **GPT** | **必须独立**（Context Contract） | 同一蓝图在不同资源条件下需要不同裁剪，这不是蓝图也不是约束 |
| **Gemini** | **必须独立**（Intent Router） | 多个蓝图如何选择？需要意图匹配层 |
| **Grok** | 未单独提出 | 认为两层 + 验收协议足够 |
| **Claude** | 未单独提出 | 认为 context_acquisition 作为附属指令足够 |

**裁决**：GPT 和 Gemini 的担忧有道理。当蓝图库扩大后，"选哪个蓝图"和"怎么裁剪"确实是真实问题。但在第一期（蓝图数量 <20），这个问题不紧迫。建议：第一期作为 context_acquisition 指令块处理，预留升级为独立层的空间。

### 8. 最终应该几层

| 模型 | 建议层数 | 结构 |
|------|---------|------|
| **Claude** | 2 层 | Blueprint + Constraints（标签维度） |
| **Grok** | 3 层 | Blueprint + Constraints + Verification Protocol |
| **GPT** | 4 部分 | Blueprint + Constraints + Context Contract + Acceptance Checks |
| **Gemini** | 4 象限 | Intent + Blueprint + Constraints + Verification |

**裁决**：见下方最终方案。

## 四、独特洞察（仅一个模型提出但有价值）

### GPT：约束有"全局"和"局部"之分
不是所有约束都挂在某个模块上。"任何回测结果都不能隐含未来函数"是全局约束，不属于任何特定阶段。

### GPT：蓝图应包含"可替换点"
明确标注哪些模块可以替换（数据源、回测器、报告器），这比只列阶段更有用。

### Gemini：蓝图必须支持"残缺注入"
用户可能已有部分代码，蓝图不能要求"推倒重来"。需要支持作为局部设计模式补丁式注入。

### Grok：蓝图需要"默认资源映射"
蓝图应标注每个阶段的默认工具选项（如数据源默认 yfinance/ccxt），让 AI 有起点可选。

### Gemini：护城河不在蓝图，在"挂在蓝图精确位置的高信噪比约束"
蓝图模式有限（架构套路就那些），但从海量 GitHub Issue 中提取的约束精准挂到蓝图节点上——这个数据飞轮才是真正壁垒。

## 五、最终裁决：种子晶体架构

综合四方意见，种子晶体的**存储模型**和**输出格式**应分开设计：

### 存储模型（知识库中怎么存）

```
蓝图（Blueprint）
  - 独立存储，一个蓝图一个文件
  - 包含：阶段划分、模块接口、数据流、可替换点、默认资源映射
  - 不包含具体实现代码
  - 标注：来源项目、适用场景、取舍说明

约束（Constraint）
  - 继续用 judgment JSONL 存储
  - 合并原知识/资源/经验三层为统一格式
  - 新增字段：applies_to（附着到蓝图的哪个阶段/模块，或 "global"）
  - 保留字段：confidence、scope、severity、evidence_refs
  - 去掉字段：layer（改为 scope + confidence 组合替代）
```

### 输出格式（种子晶体长什么样）

种子晶体 = 编译产物，由蓝图 + 约束编译而成，四段结构：

```
1. Blueprint（蓝图）
   - 系统阶段划分和数据流
   - 核心模块接口定义
   - 可替换点标注
   
2. Constraints（约束）
   - 全局约束（跨阶段）
   - 局部约束（按阶段分组挂载）
   
3. Acceptance Criteria（验收标准）
   - 从蓝图+约束交叉碰撞自动派生
   - FATAL 级约束自动提升为验收项
   
4. Context Acquisition（上下文契约）
   - 宿主 AI 必须采集的用户信息
   - 资源不足时的降级策略
   - 预留：未来升级为蓝图选择器
```

### 与旧方案的对比

| 维度 | 旧方案（三层 judgment） | 新方案（蓝图 + 约束） |
|------|----------------------|---------------------|
| 知识分类 | 按来源（知识/资源/经验） | 按功能（结构/规则） |
| 架构知识 | 缺失 | 蓝图独立存储 |
| 规则知识 | 三层重复 | 统一约束层，标签区分 |
| 验收标准 | 编译器启发式生成 | 从蓝图×约束交叉派生 |
| 上下文适配 | 固定模板 | 预留蓝图选择器升级空间 |

## 六、第一期行动建议

### 立即做
1. 设计 Blueprint 的数据模型（schema）
2. 约束层合并旧三层，新增 applies_to 字段
3. 手动提取 2-3 个蓝图作为验证（finance + ai_tooling）

### 第一期不做（预留）
1. 蓝图自动提取（人工+LLM 半自动即可）
2. Intent Router / 蓝图选择器（蓝图数量 <20 时手动选择）
3. 跨项目蓝图合成（先做单项目蓝图）

### 需要 CEO 决策
1. 蓝图的粒度标准：偏向"接口+数据流"（GPT/Gemini 建议）还是"最小可运行代码"（Grok/Claude 倾向）？
2. 约束层是否保留 layer 字段做向后兼容，还是彻底去掉？
