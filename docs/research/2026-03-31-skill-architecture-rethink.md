# 我们是不是一直在干错的事？

> 日期: 2026-03-31
> 触发: Telegram 真实用户测试 `/dora 帮我做一个提醒喝水的工具...`
> 性质: 战略级问题，非 bug 修复

---

## 一、测试中观察到的问题

### 用户操作

```
用户: /dora 帮我做一个提醒喝水的工具，每40分钟喝一次，
      并且当特斯拉股价跌破150美元时也提醒我喝一杯威士忌
宿主: 问了两个问题（通知方式、监控频率）
用户: 确认
宿主: 工具已启动并运行正常！(PID: 37379)
```

### 问题清单

| # | 问题 | 严重度 |
|---|------|--------|
| 1 | `doramagic_compiler.py` 大概率没被执行——没有"匹配到 N 个积木"的输出 | **致命** |
| 2 | 10,028 块积木、2,459 条约束、424 条失败模式全部被绕过 | **致命** |
| 3 | 用户说"确认"但没回答两个具体问题，宿主 LLM 自己假设了答案 | 高 |
| 4 | 交付中缺少能力边界、风险警告、知识来源（SKILL.md 要求的 4 项中只满足 1 项） | 高 |
| 5 | 宿主 LLM 问"Telegram 还是系统通知"——用户就在 Telegram 里对话，这不该问 | 中 |
| 6 | 工具持久性未说明（重启后 PID 消失，提醒静默停止） | 中 |
| 7 | 无知识溯源（产品宪法第 4 条被违反） | 高 |

**问题 1 和 2 是致命的：Doramagic 的核心价值（知识积木编译）被完全跳过了。宿主 LLM 用自己的知识直接生成了代码。**

## 二、对比 OpenClaw 技能规范

### 技能的本质是什么？

通过研究 OpenClaw 的技能运行时模型和 gstack 的成功技能，一个残酷的事实浮出水面：

**OpenClaw 技能是纯粹的"提示词注入"——不是可执行程序。**

| 方面 | 我们以为的 | 实际上的 |
|------|-----------|---------|
| SKILL.md | 流水线控制器，宿主 LLM 必须按步骤执行 | 建议性文档，宿主 LLM 自主决定是否遵循 |
| 脚本调用 | 强制执行（不跑脚本就不能进下一步） | 完全可选（宿主 LLM 觉得自己能搞定就跳过） |
| 约束 | 运行时强制（违反约束 → 拒绝交付） | 仅文本层面（宿主 LLM 读到但可以忽略） |
| 质量关卡 | 60 分及格线，不合格不交付 | 不存在于运行时——只在脚本内部生效，但脚本可以被跳过 |

**一句话：我们在一个"建议性平台"上建了一个"强制性流水线"。流水线的每一步都可以被宿主 LLM 跳过。**

### 技能能控制什么？不能控制什么？

**能控制的：**
- 文档化流程（写在 SKILL.md 里的步骤）
- 声明依赖（需要什么工具和环境变量）
- 提供约束文本（constraint_prompt）
- 标注能力边界

**不能控制的：**
- 不能强制宿主 LLM 执行任何脚本
- 不能阻止宿主 LLM 用自己的知识替代积木知识
- 不能在运行时验证宿主 LLM 是否遵循了约束
- 不能控制宿主 LLM 的输出格式和交付内容

## 三、成功技能是怎么做的？

### gstack 的防御模式

gstack 的技能（`/ship`、`/investigate`、`/qa`、`/careful`）面对同样的平台限制，但它们工作得很可靠。关键差异：

#### 模式 1：Hooks 代替指令

```yaml
# /investigate 技能
hooks:
  PreToolUse:
    - matcher: "Edit"
      command: "bash freeze/check-freeze.sh"
```

**不是在 SKILL.md 里写"请不要编辑冻结区域的文件"，而是用 hook 拦截编辑操作。** 这是代码级强制，不是文本级建议。

#### 模式 2：前置状态检查

```bash
# /ship 技能的 preamble
_BRANCH=$(git branch --show-current)
_PROACTIVE=$(gstack-config get proactive || echo "true")
```

**在执行任何逻辑之前，先读取系统状态。** 不假设宿主 LLM 理解了上下文。

#### 模式 3：最小化宿主 LLM 的决策负担

gstack 每个技能的 SKILL.md 给宿主 LLM 的任务是**一件具体的事**：
- `/ship` → 检查 diff、跑测试、创建 PR
- `/qa` → 打开浏览器、检查页面、报告问题
- `/careful` → 在破坏性命令前弹出警告

**没有一个 gstack 技能要求宿主 LLM 理解"5 种输入路由 → 7 个阶段 → 质量关卡 → 降级处理"这种复杂度的流程。**

#### 模式 4：输出验证循环

```
执行动作 → 解析输出 → 验证输出 → 失败则重试（最多 3 次）
```

不是"执行完信任结果"，而是"执行完验证结果"。

### Doramagic vs gstack 的根本差异

| 维度 | Doramagic | gstack 技能 |
|------|-----------|------------|
| 对宿主 LLM 的要求 | 理解并执行 7 阶段流水线 | 做一件具体的事 |
| 关键步骤强制性 | 写在 SKILL.md 里（可被跳过） | 用 hooks 拦截（代码强制） |
| 状态管理 | 依赖宿主 LLM 按顺序推进 | 前置检查 + 文件状态 |
| 失败处理 | 降级路径（但宿主 LLM 可能不进入） | 输出验证循环 |
| 复杂度 | 极高（SKILL.md 230 行） | 低-中（每个技能做一件事） |

## 四、核心诊断：我们在干什么错事？

### 错误 1：把"建议性平台"当"执行性平台"用

我们花了大量精力构建：
- 7 阶段 DAG 流水线
- 5 维度质量关卡
- 10,028 块知识积木
- DSD 虚假来源检测
- 记忆系统

**但这些全部运行在 `doramagic_compiler.py` 内部。** 而这个脚本的执行完全取决于宿主 LLM 是否"愿意"运行它。

Telegram 测试证明：**宿主 LLM 不愿意。它觉得自己能直接搞定，跳过了整个流水线。**

### 错误 2：SKILL.md 复杂度超出宿主 LLM 的可靠执行范围

Doramagic 的 SKILL.md 有 230 行，包含：
- 3 种模式（编译、提取、状态查询）
- 5 步编译流程
- 错误处理表
- 禁止行为列表
- 语言匹配规则
- 品牌语气规则

**这对宿主 LLM 来说太复杂了。** 复杂指令 → 宿主 LLM 走捷径 → 跳过关键步骤。

gstack 的做法是：每个技能只做一件事，SKILL.md 短小精悍。

### 错误 3：知识积木的价值没有在正确的层面发挥

10,028 块积木是真正有价值的资产。但它们的价值发挥路径是：

```
积木 → doramagic_compiler.py 匹配 → constraint_prompt → 注入宿主 LLM → 宿主 LLM 生成代码
```

这条路径有 4 个环节，每个环节都可能断裂。最脆弱的一环是第一步：**宿主 LLM 可能根本不调用 `doramagic_compiler.py`。**

## 五、gstack 技能给我们的启示

gstack 技能成功的模式可以总结为一个公式：

**可靠的技能 = 最小化宿主 LLM 决策 + 代码层面强制关键步骤 + 一个技能只做一件事**

具体来说：

1. **Hooks 而非指令** — 关键步骤用 `PreToolUse` hook 拦截，不写在 SKILL.md 里"请执行"
2. **前置验证** — 在做任何事之前，读取状态文件确认前置条件
3. **单一职责** — `/ship` 只管发布，`/qa` 只管测试，`/review` 只管审查
4. **输出验证** — 每一步的输出都验证后才进入下一步
5. **最低复杂度** — 宿主 LLM 需要理解的东西越少，执行越可靠

## 六、需要回答的根本问题

### 问题 1：Doramagic 应该是一个技能还是一个服务？

当前：Doramagic 是一个 OpenClaw 技能（SKILL.md + 脚本）。

但它的实际需求是：
- 多阶段流水线（7 个阶段）
- 质量控制（5 维度关卡）
- 状态管理（检查点恢复）
- 后台运行（异步提取）

**这些需求超出了"技能"的能力边界。** 一个技能能做到的上限是"给宿主 LLM 一段指令和一些脚本"。

**可能的方向：** Doramagic 是否应该变成一个独立的 MCP Server / API 服务，而不是一个被宿主 LLM "建议性执行"的技能？

### 问题 2：如果保留技能形态，应该如何重构？

参照 gstack 的成功模式：

**方向 A — 拆分为多个单职责技能**
```
/dora         → 只做一件事：用户说需求 → 匹配积木 → 注入约束 → 宿主 LLM 生成代码
/dora-extract → 只做一件事：给 GitHub URL → 提取灵魂 → 交付 SKILL.md
/dora-status  → 只做一件事：查询异步任务状态
```

**方向 B — 用 hooks 强制关键步骤**
```yaml
hooks:
  PreToolUse:
    - matcher: "Write"
      command: "bash check-bricks-matched.sh"
      # 如果还没跑过 doramagic_compiler.py，阻止代码生成
```

**方向 C — 前置自动执行，而非等宿主 LLM 调用**

当前模式：
```
宿主 LLM 读 SKILL.md → 决定是否调用脚本 → 可能跳过
```

改为：
```
/dora 被触发 → 自动执行 doramagic_compiler.py → 结果注入上下文 → 宿主 LLM 基于结果工作
```

技能触发时**自动**运行积木匹配，不给宿主 LLM"跳过"的机会。

### 问题 3：知识积木的价值应该在哪个层面发挥？

当前：积木在脚本内部匹配，结果通过 `constraint_prompt` 文本注入宿主 LLM。

**替代思路：积木直接成为宿主 LLM 的上下文**

不是"运行脚本 → 返回约束文本 → 注入宿主 LLM"，而是"积木内容直接嵌入 SKILL.md 或通过 MCP 工具返回"，让宿主 LLM 在生成代码时**自然看到**这些约束和失败模式。

这样就不存在"宿主 LLM 跳过脚本"的问题——因为积木已经在它的上下文里了。

### 问题 4：10,028 块积木的真正用法是什么？

一个让人不舒服的事实：**宿主 LLM（Claude Sonnet / GPT-4o）自己的训练数据已经包含了大量"如何写提醒工具""如何调用股票 API"的知识。**

积木的差异化价值不在"怎么做"（HOW），而在：
- **失败模式**（29% 的积木）——"别人这样做然后炸了"
- **设计理由**（18%）——"为什么要这样做而不是那样"
- **约束**（8%）——"这个 API 有速率限制，必须加退避"

**问题：** 我们是否应该把积木的定位从"编译素材"变为"审查清单"？

不是"基于积木生成代码"，而是"宿主 LLM 自己生成代码，然后用积木审查代码质量"。

---

## 七、初步判断

**是的，我们可能在干一件根本方向上有问题的事。**

具体来说：

1. **把"建议性平台"当"执行性平台"用** — 在一个宿主 LLM 可以自由跳过任何步骤的平台上，建了一个要求严格按步骤执行的 7 阶段流水线

2. **复杂度放错了地方** — 复杂度应该在代码层面（hooks、自动执行），而不是在 SKILL.md 层面（要求宿主 LLM 理解 230 行指令）

3. **知识积木的价值发挥路径太脆弱** — 积木 → 脚本 → constraint_prompt → 宿主 LLM，四步链条，任何一步断裂（尤其是第一步"宿主 LLM 不调用脚本"）就全部失效

**但核心资产（10,028 块积木、失败模式、设计理由）是真正有价值的。问题不是"做什么"，而是"怎么让它到达用户"。**

## 八、下一步建议

这不是一个 bug 修复问题，是产品架构问题。建议：

1. **立即** — 研究 gstack 的 hooks 机制，看能否用 `PreToolUse` 强制积木匹配步骤
2. **本周** — 原型测试"方向 C"（触发时自动执行，不等宿主 LLM 决定）
3. **深入** — 评估 MCP Server 方案，让 Doramagic 成为一个宿主 LLM 可以"调用"的工具，而非"阅读"的文档
4. **产品层面** — 与 CEO 讨论：积木的定位是"生成素材"还是"审查清单"？

这些方向需要 CEO 判断，因为涉及产品定位而非纯技术选择。
